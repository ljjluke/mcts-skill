# MCTS-TD 通用决策规则

> 一套模拟人脑"先想清楚再做"的推演式决策框架。
> **适用于 Claude Code。**

---

## 一句话

先理解需求 → 多轮头脑风暴想方案 → 每个方案独立推演 → 汇总决策。不脑补需求、不假装懂技术、不重复查同一份资料。

---

## 三引擎决策流程

```
用户需求理解（获取初始描述）
    │
    ▼
┌───────────────────────────────────────────┐
│  发散引擎 — 头脑风暴（可多轮循环）           │
│                                            │
│  发散 → 发现缺信息 → 问用户 → 获得新信息    │
│  → 再来一轮头脑风暴（可重复多轮）           │
│  → 收敛 → 输出方案清单                      │
│                                            │
│  产出: 方案清单（必须输出，不可跳过）        │
└───────────────────┬───────────────────────┘
                    │
                    ▼
┌───────────────────────────────────────────┐
│  推演引擎 — 子 agent 并行推演              │
│                                            │
│  子 agent A ⟳ 子 agent B ⟳ 子 agent C     │
│  （互不知道对方存在）                       │
│  独立推演 → 发现缺信息 → 记录问题 → 继续    │
│  ★ 问用户不阻塞其他子 agent                 │
│                                            │
│  产出: N份推演报告（N=方案数，不可跳过）     │
└───────────────────┬───────────────────────┘
                    │
                    ▼
┌───────────────────────────────────────────┐
│  仲裁引擎 — 汇总与决策                      │
│                                            │
│  汇总推演结果 + 汇总问题 → 统一问用户      │
│  获取答案 → 重新评估 → 执行前自检           │
│  → 输出最优方案 + 执行计划                  │
│                                            │
│  产出: 决策报告（必须输出，不可跳过）        │
└───────────────────────────────────────────┘
```

## 验证规则

确保 agent 不会跳过核心步骤：

```
检查项:
  □ 是否有"发散阶段"和"收敛阶段"的记录？ → 无=发散引擎被跳过
  □ 推演报告数量是否等于方案数量？ → 少=推演引擎被跳过
  □ 是否有决策报告？ → 无=仲裁引擎被跳过
  ✅ 三个都有 → 流程完整执行
```

### 1. 用户需求理解

先听懂用户说了什么，不急着列方案：

```
① 理解用户的初始描述:
   "帮我实现用户登录" → 认证功能需求
   
② 区分"已知"和"未知":
   已知: 用户明确说要做登录功能
   未知: 技术栈、约束条件、现有框架
   
③ 产出: 需求理解摘要（一句话说明用户要什么）
```

### 2. 发散引擎 — 头脑风暴

先发散再收敛，**过程中发现缺信息就问用户，获得新信息后可以再来一轮**：

```
① 发散阶段:
   从多个角度想可能的方案（技术/架构/资源/时间）
   至少3个，不设限制，允许天马行空

② 缺口检测:
   检查当前信息是否足够评估这些方案
   如果不够 → 问用户 → 获得新信息 → 回到①再来一轮
   
③ 收敛阶段:
   用约束过滤：违反硬约束淘汰、无依据淘汰、符合约束保留

④ 产出:
   头脑风暴清单（发散阶段的全部方案 + 收敛后的保留方案）
```

### 3. 推演引擎 — 子 agent 独立推演

每个方案由独立的子 agent 推演，**子 agent 之间互不知道对方存在**：

```
子 agent 收到的指令:
  "你只需要评估当前这个方案是否可行。
   你不知道其他方案的存在，不需要比较，不需要选最优。
   推演过程中如果发现信息不够，记录下来就行，
   不影响你继续推演。"

每个子 agent 的输出（推演报告）:
  Step 1: 起点动作 — 第一步做什么？
  Step 2: 关键路径 — 会遇到什么风险？
  Step 3: 终点评估 + 项目匹配度
  评分: V_final = V×0.6 + M×0.4
  待解决问题: [推演中发现的缺信息]

★ 重点: 子 agent 问用户不阻塞其他子 agent 的运行
```

### 4. 仲裁引擎 — 汇总与决策

```
① 汇总所有子 agent 的推演报告
② 汇总所有子 agent 记录的待解决问题
③ 统一问用户（一次问完，不是零散地问）
④ 用户回答后，重新评估受影响方案
⑤ 执行前自检（自我质疑推演结果）
⑥ 输出决策报告（最优方案 + 执行计划）
```

### 5. 执行后

```
执行 → 总结经验 → 记录到知识图谱 → 下次参考
```

## 知识召回（人脑联想机制）

```
① 联想回忆: 看到问题，最相关的知识自然浮现
② 碎片补全: 回忆不完整的，顺着线索补全
③ 外部求证: 补不全就去查资料/问用户
④ 记忆分层: active（当前意识）+ archive（长期记忆）
```

## 知识状态机

```
HYPOTHESIS → PROVISIONAL → CONFIRMED → DISPUTED → REFUTED(支持回滚)
```

## 熔断机制

```
推演准确率 ≥ 70%: 正常模式
           < 70%: 降级为简化推演
           < 50%: 暂停推演，直接问用户
```
