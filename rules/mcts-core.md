# MCTS 核心引擎 — 逐方案独立推演

## 概述

本引擎实现了 MCTS 的核心思想，但做了关键改造——**Simulation 阶段不再是快速随机模拟，而是每个方案一轮独立的结构化深度推演**。

核心设计来自 tetris_mcts 项目的三个创新：
1. **CLT-UCB 选择**（汇总比较阶段使用）
2. **Welford 在线方差**（推演置信度量化）
3. **状态投影去重**（跨会话知识复用）

---

## 引擎工作流程

```
第0步: 需求约束收集 — 问清楚再动手
第1步: 头脑风暴 — 先发散再收敛，输出方案清单
第2步: 分轮独立推演 — 每个方案由独立的子 agent 推演，输出推演报告
第3步: 汇总比较 — 合并所有推演报告，CLT-UCB 排序
第3.5步: 推演结果自检 — 自我质疑
第4步: 执行 — 只执行最优方案

任何时候用户说"直接做"、"别推了"、"不用想了" → 立即中止推演，进入执行
```

---

## 第0步: 需求约束收集（先问清楚，再动手）

在开始任何方案生成之前，**必须先系统性地收集需求约束**。这是防止"做完了才发现不对"的第一道防线。

### 0.1 约束检测清单

├──────────────────────────────────────────────────────────────────┤

### 0.2 约束来源

```
约束信息从以下渠道获取（按优先级）:

1. 用户明确说的 → 直接记录
   "用户说: 不能引入外部依赖"
   → 标记为"硬约束"，不可突破

2. 项目代码推断的 → 通过读代码发现
   "项目用了gin框架，没有其他认证中间件"
   → 标记为"事实约束"，根据代码事实确定

3. 行业/技术常识 → 从技术知识中推理
   "如果项目是金融系统，审计日志可能是隐性需求"
   → 标记为"推测约束"，需要向用户确认

4. 知识图谱中的类似任务 → 从历史经验中推断
   "类似项目（K003）也有同样的政策限制"
   → 标记为"经验约束"，可参考但需确认
```

### 0.3 约束缺失的处理

```
发现约束信息不全时:

case 可以自己确认的（从项目代码/技术知识中）:
  → 自己确认后记录为"事实约束"
  case "项目用的是gin框架" → 读go.mod → 确认 → 记录
  case "MySQL版本" → 读配置 → 确认 → 记录

case 不能自己确认的（必须问用户）:
  → 暂停 → 问用户 → 得到回答 → 继续
  case "能否引入外部依赖" → 必须问用户，不能假设"应该是可以的"
  case "有没有性能要求" → 问用户，不能假设"按常规处理"

case 用户回答了但信息不完整:
  → 追问，直到约束清晰
  → "您说不能引入外部依赖，是指不能引入新的第三方库，
     还是包括现有依赖的升级也不行？"
```

### 0.4 约束对方案的影响

```
收集到的约束直接影响方案生成:

硬约束（不可突破）:
  "不能引入外部依赖" → 排除所有需要新依赖的方案
  "只能用Java 8" → 排除使用Java 17特性的方案
  
  作用: 剪枝，减少无效方案的生成

软约束（可选，但影响匹配度）:
  "最好用MySQL，但用PostgreSQL也可以"
  "性能要求不高，但别太慢就行"
  
  作用: 影响项目匹配度M的计算
  如果方案违反软约束 → 匹配度M降低
  如果方案满足软约束 → 匹配度M升高
```

### 0.5 约束变化时的处理

```
如果推演过程中发现新的约束（之前不知道的）:

1. 将新约束加入约束清单
2. 评估已有方案是否违反新约束:
   → 违反硬约束 → 该方案直接淘汰
   → 违反软约束 → 重新计算项目匹配度M
3. 如果所有方案都违反硬约束 → 回到第1.3步重新生成方案
4. 更新全局补全箱: 记录新约束的发现

示例:
  推演方案C时发现"公司有政策不能引入外部依赖"
  → 方案C(OAuth2)违反硬约束 → 淘汰
  → 方案A(gin-jwt)和方案B(自实现JWT)不受影响 → 继续
```

### 1.1 领域熟悉度检测

```
在生成方案之前，先评估对当前领域的熟悉程度：

检测问题:
  □ 这个任务涉及的核心技术栈我是否熟悉?
  □ 这个领域常见的实现模式我是否了解?
  □ 当前项目在这方面的现有实现方式我是否清楚?

判定结果:
  熟悉度 HIGH: 所有检测项都清楚 → 可以直接凭经验生成方案
  熟悉度 MED: 部分清楚 → 先查资料补全不足部分，再生成方案
  熟悉度 LOW: 大部分不清楚 → 必须先查资料理解领域，再生成方案
```

### 1.2 领域资料收集（熟悉度 LOW/MED 时执行）

```
目标: 通过外部资料理解这个领域，而不是凭感觉列方案

资料收集路径（按优先级）:
  1. 当前项目代码
     → 读项目已有代码: "这个项目现有是怎么处理这个问题的?"
     → 理解现有模式: "已有的代码结构和约定是什么?"

  2. 技术文档 / 最佳实践
     → 搜索相关技术的官方文档或最佳实践
     → 了解行业标准做法

  3. 跨领域知识图谱
     → 查知识图谱中 tags 匹配的条目
     → 即使技术栈不同，问题模式可能相通

资料收集的产出:
  "────────────────────────────
   【领域认知】
   任务: [实现用户登录功能]
   查资料结果:
    1. 项目现有框架: gin-jwt中间件 (从代码中发现)
    2. 最佳实践: JWT + Redis黑名单 (从文档中了解)
    3. 类似案例: K003(FEATURE|API|MED) 曾用同样的模式
   领域理解:
    - 用户登录的核心流程是: 验证凭证 → 生成token → 返回
    - 安全考虑: token过期、刷新、撤销
   ────────────────────────────"
```

### 1.3 头脑风暴 — 方案发散与收敛

在领域认知的基础上，**先不受约束地发散，再用约束过滤收敛**。这是模拟人脑"先想各种可能，再考虑可行性"的过程。

```
头脑风暴规则:

① 发散阶段（不受约束，只管想）:
  从不同角度思考可能的方案:
  - 从技术角度: 用什么技术实现？
  - 从架构角度: 放在哪一层实现？
  - 从资源角度: 自己做还是用现成的？
  - 从时间角度: 一次性做完还是分步做？
  
  发散要求:
  - 至少想出 3 个方案，上不封顶
  - 不要在这个阶段判断"这个方案行不行"
  - 允许不成熟的想法、天马行空的想法

② 收敛阶段（用约束过滤）:
  用第0步收集的约束清单过滤:
  - 违反硬约束的 → 淘汰，注明原因
  - 违反软约束的 → 保留，标记匹配度影响
  - 没有依据的 → 淘汰，注明"无依据"
  - 符合约束且有依据的 → 保留

③ 输出方案清单（必须输出，不可跳过）:
```

#### 可验证产出：头脑风暴清单

```
═══════════════════════════════════════
  头脑风暴清单
═══════════════════════════════════════

发散阶段（共想出 N 个方案）:
  1. [方案名称] — [一句话描述]
  2. [方案名称] — [一句话描述]
  3. [方案名称] — [一句话描述]
  ...

收敛阶段（用约束过滤）:
  □ [方案A] → 保留 (符合约束，依据:XXX)
  □ [方案B] → 淘汰 (违反硬约束:不能引入外部依赖)
  □ [方案C] → 保留 (符合约束，依据:XXX)
  □ [方案D] → 淘汰 (无依据，凭空想的)
  ...

最终保留方案（至少2个，最多5个）:
  1. [方案A] — [一句描述+依据]
  2. [方案C] — [一句描述+依据]
  ...

═══════════════════════════════════════

⚠️ 验证: 如果输出中没有"发散阶段"和"收敛阶段"两步说明 → 说明跳过了头脑风暴
```

### 1.4 粗筛（仅当方案数 > 5 时触发）

```
粗筛方法: 对每个方案做一次快速直觉评估

评估维度:
  - 可行性: 这个方案在当前上下文中是否可行? (0-1)
  - 成本效益: 投入产出比是否合理? (0-1)
  - 风险: 引入新问题或破坏现有功能的可能性? (0-1)

粗筛公式:
  score = 可行性 × 0.5 + 成本效益 × 0.3 + (1 - 风险) × 0.2

保留 top-3~5 个方案进入深度推演阶段。
被淘汰的方案记录在"备用列表"中，如果主方案全部不可行则重新启用。
```

## 第2步: 分轮独立推演（核心机制）

### 核心原则

```
★ 强制规则: 每个方案必须在独立上下文中完成推演，不能在同一上下文中
   假装"分开思考" ★

这意味着:
  - 方案A的推演不能知道方案B的存在
  - 方案B的推演不能参考方案A的结果
  - 每个推演是"如果只做这个方案，会怎样"的独立分析

推荐实现方式:
  ✅ 子 agent 模式（推荐）:
     每个方案启动一个独立的子 agent，各自完成推演。
     子 agent 只收到自己的方案描述，不知道其他方案的存在。
     子 agent 独立输出推演报告，主流程汇总。

  ✅ 分段隔离模式（平台不支持子 agent 时的备选）:
     在同一对话中明确分段:
     "═══ 开始方案A的独立推演 ═══" → 推演 → "═══ 方案A结束 ═══"
     → 重置上下文 → 
     "═══ 开始方案B的独立推演 ═══" → 推演 → "═══ 方案B结束 ═══"
     分段之间必须清空上一个方案的思考痕迹。

  ❌ 禁止的"假推演"行为:
     × 在同一段思考中列出所有方案的优缺点再比较
     × 先说"方案A可以XX，方案B可以YY，所以选A"
     × 用"综上所述，方案A最好"代替独立推演
     
     ⚠️ 验证: 如果最终输出中没有每个方案的独立推演报告，
        说明跳过了真正的推演步骤。

★ 用户可以随时中止:
  在任何阶段，用户说"直接做"、"别推了" → 立即中止，进入执行
```

### 子 agent 推演指令模板

如果平台支持子 agent（如 Claude Code 的 Agent 工具），每个方案的推演发给子 agent 的指令必须包含以下内容，**确保子 agent 不知道其他方案的存在**：

```
你收到的是一个独立的推演任务。你的目标是：
  1. 你不知道其他方案的存在，你只需要评估当前这个方案
  2. 你不需要比较，不需要选最优
  3. 你只需要回答"这个方案本身行不行"

方案描述:
  [方案名称]: [一句话描述]
  依据: [来源]
  约束: [必须遵守的约束]

请按以下格式完成推演:
  Step 1: 起点动作 — 第一步做什么？
  Step 2: 关键路径 — 会有什么风险？
  Step 3: 终点评估 + 项目匹配度
  评分: V=?, M=?, V_final=?

完成后输出标准推演报告（格式见下文）。
```

### 推演框架（详细规则）

在开始每个方案的推演之前，**必须执行完整的多路召回流程**，从知识图谱中获取相关知识：

#### 召回执行步骤

```
步骤1: 确定当前查询向量
  特征向量: 从当前任务提取的六维特征
  上下文: {tech_stack, framework, project_type}
  方案关键词: 从方案描述中提取的关键词列表（用于标签匹配）

步骤2: 执行五路召回
  路径1: 精确匹配 → 基于六维特征向量
  路径2: 主维度匹配 → 基于 task_type + domain
  路径3: 领域匹配 → 基于 domain + risk_level
  路径4: 标签匹配 → 基于方案关键词 vs 知识条目 tags
  路径5: 关联扩散 → 基于已召回条目的关联关系

步骤3: 合并评分排序
  去重 → 按综合评分排序 → 取 top-5

步骤4: 输出召回报告
  "召回报告: [具体引用的知识条目K00X, K00Y]
   路径来源: [精确匹配/标签匹配/...]
   冲突提示: [如果有矛盾条目，输出冲突信息]
   引用建议: [推荐优先参考哪条]"
```

#### 知识注入到推演

```
根据召回结果，按以下方式注入推演：

case 召回 top-5 中最高分为 CONFIRMED 且 score > 0.8:
  → 作为主参考知识
  → Step 1: "根据历史经验(K00X)，这类任务的典型路径是..."
  → 推演可以紧密参考该知识的上下文

case 召回 top-5 中存在技术栈匹配的知识:
  → 优先参考技术栈匹配的知识（即使它是PROVISIONAL）
  → Step 2: "当前技术栈下，参考K005的经验..."

case 召回 top-5 中存在冲突:
  → 输出冲突提示
  → 推演中分别引用两个方向，注明"历史经验有分歧，需特别注意"
  → Step 3: 最终价值以当前推演为主，历史价值仅作参考

case 召回结果为空或最高分 < 0.2:
  → "无相关知识，冷启动推演"
  → 完全依赖当前分析
  → 推演完成后创建新的HYPOTHESIS条目
```

### ⚠️ 推演中的缺口检测与信息补全（关键）

人脑在推演过程中如果发现信息不全，会主动去获取信息再继续。Claude Code 不能脑补，必须**停下来、检测缺口、补全信息、再继续推演**。

#### 缺口的两种类型

```
缺口类型1: 需求/上下文信息缺失
  表现: 
    - "用户没有说用什么技术栈" → 不能假设是"最常见的那个"
    - "不知道这个项目的代码结构" → 不能假设"应该是MVC架构"
    - "不清楚用户偏好的风格" → 不能假设"用统一风格"
  
  行动: 暂停推演 → 向用户提问 → 得到回答 → 继续推演

缺口类型2: 技术知识不够
  表现:
    - "这个API的用法我不确定" → 不能自己编一个用法
    - "这个框架的某个特性我没用过" → 不能假设"和别的框架一样"
    - "这个错误信息的含义不清楚" → 不能猜测
  
  行动: 暂停推演 → 查资料(内置知识/技术文档/代码库) → 搞懂后继续推演
```

#### 规则1：缺口检测（推演每一步完成后必须检查）

```
每完成一个推演步骤后，必须执行缺口检测：

检测清单:
  □ 这个步骤依赖的上下文信息是否都有？
    如果有缺失 → 标记为"需求缺口"
    "推演Step 1完成，但发现不知道项目使用什么数据库，无法评估方案A的影响范围"
    → 暂停推演，执行信息补全

  □ 这个步骤涉及的技术方案是否都理解？
    如果有不懂的 → 标记为"技术缺口"
    "推演Step 2发现需要用到Kubernetes的StatefulSet，但不确定其PVC回收策略"
    → 暂停推演，执行信息补全

  □ 这个步骤的推演是否依赖了"假设"？
    如果是 → 标记为"假设缺口"
    "假设了用户的API都遵循RESTful规范"
    "假设了数据库版本是MySQL 8.0+"
    → 如果假设有50%以上概率不成立，暂停推演并提问确认
```

#### 规则2：信息补全行动

```
根据缺口类型执行不同的补全行动：

需求缺口 → 向用户提问:
  格式:
    "────────────────────────────
     【需求补充】
     推演[方案A]时发现信息缺失:
     问题: [具体什么问题]
     影响: [不补充这个信息，推演会产生XX偏差]
     请补充: [用户需要提供的具体信息]
     ────────────────────────────"
  
  要求:
    - 必须问具体问题，不能笼统问"还有什么信息"
    - 一次最多问2个问题（问太多用户记不住）
    - 用户回答后继续推演
    - 如果用户说"你决定" → 标记为"用户授权假设"，继续推演但降低信心

技术缺口 → 自动查资料:
  行动路径（按优先级）:
    1. 利用Claude的内置知识（模型训练数据中的技术知识）
    2. 搜索相关技术文档（通过WebSearch或read-apidocs工具）
    3. 读取项目中的现有代码（如果有，了解已有模式）
  
  格式:
    "────────────────────────────
     【技术补充】
     推演[方案A]时发现知识不足:
     主题: [什么技术点]
     问题: [具体不清楚什么]
     查资料结果: [查到的事实]
     对推演的影响: [修正了之前的哪个假设]
     ────────────────────────────"
  
  要求:
    - 必须明确说出"我之前不确定XX，查了资料后发现YY"
    - 不能只看一篇资料就下结论，至少2个来源交叉验证
    - 如果查不到 → 标记为"高风险假设"，推演中方差+0.2

假设缺口 → 根据置信度处理:
  ┌─ 置信度 ≥ 80%: 继续推演，在报告中标记"假设:XXX"
  ├─ 50% ≤ 置信度 < 80%: 向用户确认
  └─ 置信度 < 50%: 必须向用户确认，不能继续
```

##### 补全箱查询规则

```

  1. 用"主题关键词"搜索全局补全箱

     → "K8s持久化存储" → 匹配G002（StatefulSet PVC）
     → 确认是同一个知识点，直接引用

     → 完成后写入全局补全箱

  4. 补全箱中已有但状态是"待处理":

```

##### 补全后更新规则

```

  方案B需要G002 → 查全局箱 → 已存在 → 直接引用
  方案C需要G002 → 查全局箱 → 已存在 → 直接引用

  效果:
    - 1次查资料，3个方案受益
    - 随着方案增多，边际成本递减
```

##### 补全箱的作用域

```
全局补全箱只在当前决策会话中有效。

跨会话共享的是"知识图谱"（td-learner.md），
不是"临时补全箱"。
```

##### 补全箱与知识图谱的关系

```
信息补全箱（临时）            知识图谱（持久）
─────────────────────────────────────────────────

```

#### 规则4：补全后对推演的修正

```
补全信息获得后，必须明确修正推演：

修正规则:
  "补全前 -> 补全后"

  如果补全的信息与原推演假设一致:
    → 推演结果不变，信心提升
    → Step X标注: "假设已确认(来源:用户/资料)，信心从MED→HIGH"

  如果补全的信息与原推演假设矛盾:
    → 推演结果必须修正
    → Step X标注: "假设修正(原假设:XX, 修正为:YY)，推演结果调整"
    → 如果修正幅度 > 0.3，需要重新评估后续步骤

  如果补全的信息补充了原有缺失:
    → 推演可以更深入
    → 在修正后的基础上继续推演后续步骤
```

#### 规则5：防止错误的两种保护机制

```
保护1: 禁止自动补全需求
  任何时候，如果推演需要"用户的意图/偏好/约束"， 
  不能自己脑补 → 必须暂停推演 → 问用户
  例外: 用户明确说了"你决定"

保护2: 对外查资料不能只信一个来源
  查技术资料时:
    - 至少2个独立来源交叉验证
    - 如果2个来源矛盾 → 标记为"有争议的技术点"
    - 如果只有1个来源 → 标记为"单一来源，谨慎使用"
    - 没有来源 → 标记为"未知领域"，推演方差增加
```

#### 第一步：评估价值函数的可信度

在引用任何历史价值之前，先评估其可信度：

```
价值函数可信度评估:

  1. 访问次数 (n):
     n < 5  → 数据太少，不可信，仅作参考
     5 ≤ n < 20 → 有一定参考价值，但不可盲信
     n ≥ 20 → 统计上可信

  2. 方差 (σ²):
     σ² < 0.05 → 高度一致，可信
     0.05 ≤ σ² < 0.2 → 正常波动，可参考
     σ² ≥ 0.2 → 结果不稳定，需要谨慎

  3. 时效性:
     24小时内更新 → 新鲜
     1周内更新 → 较新
     超过1个月 → 可能过时

  4. 上下文匹配度:
     历史状态的六维特征与当前状态完全匹配 → 高
     部分匹配 → 中
     低匹配 → 低

综合可信度 = 
  LOW(低)  : 仅作灵感参考，不用于评分
  MED(中)  : 作为推演锚点，但保留质疑权
  HIGH(高) : 可信，可作为价值修正的依据
```

#### 第二步：价值注入方式

根据可信度决定注入方式：

```
综合可信度 HIGH:
  → 将历史价值作为推演的"预期锚点"
  → Step 1: "历史数据显示这类任务成功率XX，本次推演以此为基准"
  → Step 3: 最终价值 = 推演价值 × 0.5 + 历史价值 × 0.5

综合可信度 MED:
  → 将历史价值作为"参考基线"
  → Step 1: "历史数据显示XX，但需要结合当前上下文验证"
  → Step 3: 最终价值 = 推演价值 × 0.7 + 历史价值 × 0.3

综合可信度 LOW:
  → 仅将历史价值作为"背景参考"，不作为评分依据
  → "过去有类似案例但数据不足，本次推演主要依赖当前分析"
  → Step 3: 最终价值 = 推演价值 × 1.0（不引用历史）
```

#### 第三步：推演中发现价值函数不准时的处理

如果在推演过程中发现**历史价值与当前推理明显矛盾**，不能忽视，必须做纠偏：

```
推演中检测到矛盾的触发条件:
  条件A: 推演Step 1的结果与历史价值方向相反
         (历史说成功率80%，但推理发现当前上下文有根本性差异)
  条件B: 推演Step 2发现了历史数据中不存在的风险
         (技术栈变了/依赖更新了/项目结构改了)
  条件C: 推演的每个步骤都显示历史价值明显偏高/偏低
         (连续3步推理结果都与历史价值偏离 > 0.3)

触发任何一条时，执行以下纠偏流程:
  
  1. 暂停推演，输出"价值函数矛盾检测":
     "历史价值: q=0.85 (基于n=30次历史数据)
      当前推理发现: [具体的矛盾点]
      矛盾原因分析: [为什么历史价值不适用于当前场景]

  2. 修正价值函数使用方式:
     → 该条历史价值从HIGH降级为LOW
     → 当前推演不再依赖该历史价值
     → 推演完成后，在TD更新阶段记录这条矛盾

  3. 继续推演（基于当前分析，不依赖历史）:
     "推演继续，不引用该历史价值。
      本次推演结果将作为新基准，用于更新价值函数。"

  4. 推演结束后在报告中标记:
     "价值函数矛盾已记录: [具体条目]，将在TD更新时处理"
```

#### 第四步：分步知识注入

在每个推演步骤中，知识注入的方式不同：

```
Step 1 — 起点动作:
  → 注入: 相似任务的历史成功率
  → 检查: 当前起点与历史起点是否一致?
    • 一致 → 可信度高，价值锚点成立
    • 不一致 → 可能走入不同分支，降低历史权重

Step 2 — 关键路径:
  → 注入: 历史决策序列模式（成功路径/失败路径）
  → 检查: 当前路径是否与历史路径匹配?
    • 匹配 → 可以借鉴历史路径的应对策略
    • 不匹配 → "这是新路径"，标记为潜在矛盾

Step 3 — 终点评估:
  → 注入: 历史终点的价值分布
  → 修正: 最终价值 = 推演价值 × w1 + 历史价值 × w2
  → 权重 w1/w2 由前两步的矛盾检测结果决定
    • 没矛盾: w1=0.5, w2=0.5
    • 部分矛盾: w1=0.7, w2=0.3
    • 明显矛盾: w1=1.0, w2=0.0（弃用历史）
```

## 第3步: 汇总比较

### 汇总表

所有方案推演完成后，将结果汇总到一个表中。**汇总表包含原始价值V和项目匹配度M两个维度**：

```
┌───────┬──────┬──────┬──────┬──────┬──────────────────────────────────┐
│ 方案   │ 原始V│ 匹配M│ V_final │ 方差 │ 优势/劣势与项目匹配分析         │
├───────┼──────┼──────┼──────┼──────┼──────────────────────────────────┤
│ 方案A │ 0.90 │ 0.85 │ 0.88 │ 0.05 │ 优势:性能高 ✓匹配;劣势:需改DB ✗不在意│
│ 方案B │ 0.72 │ 0.95 │ 0.81 │ 0.12 │ 优势:零依赖 ✓匹配;劣势:性能中 ✓容忍  │
│ 方案C │ 0.91 │ 0.30 │ 0.67 │ 0.08 │ 优势:功能全 ✗不需要;劣势:成本高 ✗痛点│
└───────┴──────┴──────┴──────┴──────┴──────────────────────────────────┘

解读:
  - 方案C原始价值最高(0.91)，但项目匹配度最低(0.30)
    → 因为它的优势(功能全)项目不需要，劣势(成本高)是项目痛点
    → V_final只有0.67，排名第三

  - 方案A原始价值高(0.90)，匹配度也高(0.85)
    → 优势匹配、劣势不在意 → V_final=0.88，推荐

  - 方案B原始价值中等(0.72)，但匹配度最高(0.95)
    → 虽然不是技术上最好的方案，但最适合当前项目
    → V_final=0.81，可以作为备选
```

### CLT-UCB 排序（使用 V_final）

用 CLT-UCB 公式计算每个方案的最终分数，**使用 V_final 而非原始 V**：

```
UCB = V_final + Φ⁻¹(N) × √(方差 / 1)

其中 N = 方案总数
Φ⁻¹(N) 近似值:
  N=2 → 1.5
  N=3 → 1.0
  N=4 → 0.8
  N=5 → 0.7
```

### 排序规则

```
1. 按 UCB 分数从高到低排序

2. 如果两个方案的 UCB 分数相差 < 0.05:
   → 比较项目匹配度 M: 匹配度高的优先
   → 如果 M 也接近 → 比较方差: 方差小的优先
   → 如果还接近 → 用户选择

3. 输出最终排序列表，附带每个方案的"优势/劣势 vs 项目需求"分析

4. 选择第1名作为执行方案

5. 如果第1名在V_final上领先不多(<0.05)，且第2名的匹配度显著更高:
   → 输出"建议: 考虑方案B，虽然技术评分略低，但更适合项目"
```

## 第3.5步: 推演结果自检（关键防错）

在选中方案执行之前，**必须对推演结果做一次自我质疑**。这是防止"推演错了还按错的执行"的最后一道防线。

### 为什么需要自检

```
推演过程本身也可能出错:
  - Claude 推理时漏掉了某个关键因素
  - 推演时依赖了一个错误的假设
  - 某个方案的风险被低估了
  - 价值函数中的历史数据可能已经过时

不自检就去执行 → 错了也不知道 → 错误的价值被TD学习吸收
  → 后续推演越来越不准 → 恶性循环
```

### 自检步骤

```
在"选择第1名作为执行方案"之后，执行之前，做三件事:

① 找漏洞:
  "这个推演有什么可能出错的地方？"
  → 检查推演过程中是否有模糊的判断
  → 检查是否依赖了未经验证的假设
  → 检查是否有被忽略的风险
  
② 反向思考:
  "如果第2名其实比第1名好，可能是什么原因？"
  → 强迫自己从另一个角度看问题
  → 找出第1名的盲区
  
③ 风险评估:
  "如果选错了，最坏的结果是什么？能承受吗？"
  → 如果能承受 → 执行
  → 如果不能承受 → 进入"再推演"模式，或建议用户手动确认
```

### 自检报告格式

```
═══════════════════════════════════════
  推演结果自检
═══════════════════════════════════════

  ① 找漏洞:
    □ 推演中是否有模糊的判断? → [有/无]
      如果有: [具体说明]
    □ 是否依赖了未验证的假设? → [有/无]
      如果有: [具体说明]
    □ 是否有被忽略的风险? → [有/无]
      如果有: [具体说明]

  ② 反向思考:
    如果第2名[方案B]比第1名[方案A]好，可能的原因:
    - [原因1]
    - [原因2]
    这些原因成立的可能性: [高/中/低]
    如果成立，是否改变选择? → [是/否]

  ③ 风险评估:
    选第1名[方案A]的最坏结果: [描述]
    最坏结果的影响程度: [轻微/中等/严重]
    是否能承受? → [能/不能]

  自检结论:
    ✅ 通过 — 推演可靠，可以执行
    ⚠️ 有风险 — 建议用户确认后再执行
    ❌ 未通过 — 重新推演或换方案
```

### 自检未通过时的处理

```
case 自检结论为 ⚠️ 有风险:
  → 输出自检报告
  → 建议用户确认: "我推演的结果是方案A最好，但有几个不确定的点，你要不要确认一下？"
  → 用户确认后执行
  → 用户否决后回到第2步

case 自检结论为 ❌ 未通过:
  → 输出自检报告
  → 自动回到第2步重新推演
  → 如果连续2次自检都未通过 → 输出"这个问题我不太确定，建议你手动决策"
```

### 熔断机制

```
记录推演准确率:
  每次执行后，对比推演结果和实际结果
  推演准确率 = 最近10次推演中误差 < 0.3 的次数 / 10

熔断规则:
  准确率 < 70% → 自动降级为简化推演（2步，不深入）
  准确率 < 50% → 暂停推演，改为直接问用户"你觉得哪个方案好？"
  连续3次准确率 < 50% → 输出"我可能不太擅长这类问题，建议你手动决策"

准确率恢复:
  如果后续5次推演的准确率 > 80%
  → 恢复完整推演模式
```

### 执行遇到意外时的处理

```
执行过程中如果遇到推演时未预见的问题:
  1. 记录意外: 推演时的假设是什么? 实际情况是什么?
  2. 评估影响: 这个意外是否导致当前方案不可行?
     → 如果不可行: 触发"再推演"模式
     → 如果仍可行: 继续执行，但调整推演评分
  3. 再推演: 回到第2步，对排名第2的方案做推演（如果之前没做过）
  4. 切换执行: 如果第2名更优，切换过去
```

### TD误差记录（执行中）

```
每次遇到意外时:
  TD_error = 实际结果 - 推演预期
  这个误差会被记录，在最终TD更新阶段使用
```
