# MCTS-TD Planner — 多方案独立推演决策引擎

> **核心能力**：当有多个候选方案时，每个方案都在内部独立走完完整的执行路径推演（不实际执行），然后汇总比较选出最优方案，最后才执行。

此 Skill 将 **MCTS（蒙特卡洛树搜索）** 和 **TDL（时序差分学习）** 的算法思想注入到 Claude 的推理过程中。它不是实际的数值计算引擎，而是**将算法的核心决策逻辑转化为结构化的推理规则**，让 Claude 在面对复杂决策时，能够像拥有"内部推演棋盘"一样，系统地、逐个地推演每个候选方案，再综合比较。

> 本项目受 [hrpan/tetris_mcts](https://github.com/hrpan/tetris_mcts) 启发，该项目的 MCTS-TD 混合架构在俄罗斯方块游戏中取得了超越人类水平的成绩。

## 核心决策流程

```
                    ╔══════════════════════════════════════╗
                    ║      三引擎决策流程                   ║
                    ║      每个引擎有可验证产出，不能跳过    ║
                    ╚══════════════════════════════════════╝

用户需求理解（先听懂用户说了什么）
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  发散引擎 — 头脑风暴（可多轮循环）                            │
│                                                             │
│  ① 发散: 从多个角度想方案，至少3个，不设限制                  │
│  ② 检测: 缺信息? → 问用户 → 获得新信息                       │
│  ③ 再发散: 基于新信息再来一轮（可重复多轮）                    │
│  ④ 收敛: 用约束过滤 → 保留可行方案                           │
│                                                             │
│  ⚠️ 产出: 方案清单（必须有发散和收敛的记录）                   │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  推演引擎 — 子 agent 并行推演                                │
│                                                             │
│  子 agent A（不知道B/C存在）                                  │
│  → 独立推演 → 发现缺信息 → 记录问题 → 继续推演               │
│                                                             │
│  子 agent B（不知道A/C存在）                                  │
│  → 独立推演 → 发现缺信息 → 记录问题 → 继续推演               │
│                                                             │
│  ★ 问用户不阻塞其他子 agent                                  │
│                                                             │
│  ⚠️ 产出: N份推演报告（N=方案数，少一份就是跳过）             │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  仲裁引擎 — 汇总与决策                                       │
│                                                             │
│  ① 汇总推演结果                                              │
│  ② 汇总待解决问题 → 统一问用户                                │
│  ③ 获取答案 → 重新评估受影响方案                              │
│  ④ 执行前自检 → 输出最优方案 + 执行计划                       │
│                                                             │
│  ⚠️ 产出: 决策报告（必须有自检和决策结论）                     │
└─────────────────────────────────────────────────────────────┘
```

## ⚡ 验证规则

确保三个引擎都执行了，没有跳过：

```
检查项:
  □ 是否有"发散阶段"和"收敛阶段"的记录？ → 无=发散引擎被跳过 ❌
  □ 推演报告数量是否等于方案数量？ → 少=推演引擎被跳过 ❌
  □ 是否有决策报告（含自检）？ → 无=仲裁引擎被跳过 ❌
  ✅ 三个都有 → 流程完整执行
```

## ⚡ 自动启动检测（关键）

## ⚡ 自动启动检测（关键）

本 Skill 设置为 `alwaysApply: true`，意味着每次对话都会加载。
但加载 ≠ 启动决策流程。**是否启动决策流程，由以下自动检测逻辑决定**。

### 检测时机

```
检测发生在两个时刻:

时刻1: 每次收到用户消息后
  → 判断: 用户的问题是否涉及"需要决策"?
  → 如果是 → 启动完整决策流程
  → 如果不是 → 保持静默，不干扰正常对话

时刻2: 每次工具调用返回结果后
  → 判断: 刚刚执行的结果是否表明"当前方案不行"?
  → 如果是 → 启动"再推演"模式
  → 如果不是 → 继续当前路径
```

### 时刻1的检测规则

```
收到用户消息后，执行快速检测。

核心原则: 任何任务都有多种做法，人脑会先快速想一下再决定怎么做。
          引擎模拟这个"快速想一下"的过程，但只在有多种显著不同方案时
          才启动完整流程。

检测流程（极简，一次快速自问）:

问自己: "这个问题/任务，如果用不同的方式去做，结果会不会明显不同？"

判断标准:
  ① 有多种工具/方法/路径可以达到目的？
  ② 不同选择会带来不同的后续影响？
  ③ 在这个问题上花点时间思考值不值得？

回答"是"的越多 → 启动引擎
    
示例:
  "帮我看看这个文件"
    → 可以读全部/搜关键词/只看结构 → 是①
    → 不同方式会漏掉不同信息 → 是②
    → 花几秒钟想一下怎么读值不值得 → 是③
    → 启动简化推演 ✓

  "把这个按钮颜色改成蓝色"
    → 可以改CSS/改JS变量/改主题 → 是①
    → 不同方式影响后续维护 → 是②  
    → 花几秒钟想一下值不值得 → 是③（如果项目有规范）
    → 启动简化推演 ✓

  "你好"
    → 打招呼，没有多种做法 → 否①②③
    → 不触发 ✗

  "什么是MCTS算法"
    → 信息类问题，不同解释方式都是同一个答案 → 否①
    → 不触发 ✗

触发后的行为:
  输出"⚡ [MCTS-TD] 检测到决策需求" 并启动第0步
```

### 如何理解"检测发生在时刻1"

```
不要误解: "每次收到消息都要检测一遍，会不会很慢？"

不会。检测的逻辑是:
  1. 在你的正常推理过程中，额外做一个快速判断
  2. 这个判断不需要调用任何工具，不需要查资料
  3. 只是问自己一个问题: "这任务有多个可行的方案吗？"
  4. 有 → 启动引擎；没有 → 跳过

类比人脑:
  别人让你做一件事，你下意识就会想:
  "这个嘛...有好几种做法" → 然后开始想哪种最好
  "这个很简单，直接做就行" → 直接动手
  这个Skill要模拟的就是这个"下意识判断"。
```

### 时刻2的检测规则

```
工具调用返回后，检测执行结果:

检测项A: 执行是否遇到意外?
  编译失败? → 触发"再推演" ✓
  测试失败? → 触发"再推演" ✓
  结果与预期不符? → 触发"再推演" ✓
  一切正常? → 不触发 ✗

检测项B: 执行结果是否改变了之前的判断?
  发现了之前不知道的约束? → 触发 ✓
  某个方案被证明不可行? → 触发 ✓
  新信息让其他方案变得更优? → 触发 ✓
  没有新信息? → 不触发 ✗

触发"再推演"时:
  → 记录本次执行的TD误差
  → 回到第2阶段，对剩余方案重新推演
  → 如果当前方案仍最优 → 继续执行
  → 如果有更优方案 → 切换
```

### 启动信号

```
当自动检测决定启动决策流程时，输出以下启动信号:

"═══════════════════════════════════════
 ⚡ [MCTS-TD] 检测到决策需求，启动决策引擎
 触发原因: [检测项A/B/C的具体描述]
 启动模式: [完整推演/简化推演/再推演]
═══════════════════════════════════════"

这个信号让用户知道:
  - 引擎被触发了
  - 为什么被触发
  - 当前处于哪个阶段
```

### 不干扰原则

```
当自动检测决定"不触发"时:
  → 不输出任何与引擎相关的内容
  → 完全保持静默，像不存在一样
  → 不影响正常的对话和工作流程
  
目的: 这个 Skill 是"备用决策系统"，只在需要时启动，
      不需要时完全不干扰用户。
```


### 模式一：完整模式（默认）

```
枚举方案(限5个) → 逐轮推演(每方案1轮) → 汇总比较(CLT-UCB) → 执行 → TD更新
```

### 模式二：快速推演（方案数 > 5）

```
直觉粗筛 → 保留top-3 → 逐轮推演 → 汇总比较 → 执行
```

### 模式三：再推演（执行途中遇到意外）

```
执行受阻 → 记录TD误差 → 回到第2阶段重新推演剩余方案 → 切换执行
```
