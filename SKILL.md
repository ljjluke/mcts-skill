# MCTS-TD Planner — 多方案独立推演决策引擎

> **核心能力**：当有多个候选方案时，每个方案都在内部独立走完完整的执行路径推演（不实际执行），然后汇总比较选出最优方案，最后才执行。

此 Skill 将 **MCTS（蒙特卡洛树搜索）** 和 **TDL（时序差分学习）** 的算法思想注入到 Claude 的推理过程中。它不是实际的数值计算引擎，而是**将算法的核心决策逻辑转化为结构化的推理规则**，让 Claude 在面对复杂决策时，能够像拥有"内部推演棋盘"一样，系统地、逐个地推演每个候选方案，再综合比较。

> 本项目受 [hrpan/tetris_mcts](https://github.com/hrpan/tetris_mcts) 启发，该项目的 MCTS-TD 混合架构在俄罗斯方块游戏中取得了超越人类水平的成绩。

## 三引擎决策流程

```
                    ╔══════════════════════════════════════╗
                    ║      三引擎决策流程                   ║
                    ║      每个引擎有可验证产出，不能跳过    ║
                    ╚══════════════════════════════════════╝

用户需求理解（先听懂用户说了什么）
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  发散引擎 — 头脑风暴（可多轮循环）                            │
│                                                             │
│  ① 发散: 从多个角度想方案，至少3个，不设限制                  │
│  ② 检测: 缺信息? → 问用户 → 获得新信息                       │
│  ③ 再发散: 基于新信息再来一轮（可重复多轮）                    │
│  ④ 收敛: 用约束过滤 → 保留可行方案                           │
│                                                             │
│  ⚠️ 产出: 方案清单（必须有发散和收敛的记录）                   │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  推演引擎 — 子 agent 并行推演                                │
│                                                             │
│  子 agent A（不知道B/C存在）                                  │
│  → 独立推演 → 发现缺信息 → 记录问题 → 继续推演               │
│                                                             │
│  子 agent B（不知道A/C存在）                                  │
│  → 独立推演 → 发现缺信息 → 记录问题 → 继续推演               │
│                                                             │
│  ★ 问用户不阻塞其他子 agent                                  │
│                                                             │
│  ⚠️ 产出: N份推演报告（N=方案数，少一份就是跳过）             │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  仲裁引擎 — 汇总与决策                                       │
│                                                             │
│  ① 汇总推演结果                                              │
│  ② 汇总待解决问题 → 统一问用户                                │
│  ③ 获取答案 → 重新评估受影响方案                              │
│  ④ 执行前自检 → 输出最优方案 + 执行计划                       │
│                                                             │
│  ⚠️ 产出: 决策报告（必须有自检和决策结论）                     │
└─────────────────────────────────────────────────────────────┘
```

## 验证规则

确保三个引擎都执行了，没有跳过：

- ✅ 有"发散阶段"和"收敛阶段"的记录 → 发散引擎已执行
- ✅ 推演报告数量等于方案数量 → 推演引擎已执行
- ✅ 有决策报告（含自检）→ 仲裁引擎已执行

## 三种运行模式

| 模式 | 适用场景 | 流程 |
|------|---------|------|
| **完整模式** | 方案数 ≤ 5 | 枚举方案 → 逐轮推演 → 汇总比较(CLT-UCB) → 执行 → TD更新 |
| **快速推演** | 方案数 > 5 | 直觉粗筛 → 保留top-3 → 逐轮推演 → 汇总比较 → 执行 |
| **再推演** | 执行途中遇到意外 | 记录TD误差 → 重新推演剩余方案 → 切换执行 |

## 启动规则

本 Skill 设置为 `alwaysApply: true`，每次对话都会加载。但加载 ≠ 每次都启动完整三引擎流程。

**每次收到用户消息后，Skill 强制启动「发散引擎」进行头脑风暴：**

### 第1步：需求拆解（如果用户消息包含多个子需求）

用户一句话里可能藏着多个需求，先把它们拆开：

```
用户说: "帮我做一个博客系统，支持markdown，还要有评论功能"
拆解: 
  → 需求A: 搭建博客系统框架
  → 需求B: 实现markdown支持
  → 需求C: 实现评论功能
  
然后对每个需求独立执行下面的发散流程。
```

拆解原则：**按"能不能独立选择方案"来拆**，而不是硬拆成最小粒度。

### 第2步：真实发散

对每个需求（或单个需求），从多个角度想方案：

```
怎么才算"多个角度"？
  → 不同的技术路线（React vs Vue / JWT vs Session）
  → 不同的实现策略（自己实现 vs 用库 vs 用服务）
  → 不同的设计取舍（性能优先 vs 可维护优先 vs 成本优先）
  → 不同的架构层级（前端改 vs 后端改 vs 都改）
```

**关键规则：**

| 规则 | 说明 |
|------|------|
| **不凑数** | 有多少种真实可行的方案就列多少种，1种就1种，10种就10种，不硬凑"至少3个" |
| **不虚构** | 每一种方案都必须是你实际会去执行的，不能为了数量编造不存在的做法 |
| **不遗漏** | 如果确实存在多种合理方案，不要因为"麻烦"而只列1种 |
| **一需求一发散** | 多个子需求时，每个子需求独立发散，独立选择方案 |

### 第3步：走哪条路？

| 头脑风暴结果 | 行为 |
|-------------|------|
| **≥2 个可行方案** | ✅ 进入推演引擎，启动推演→仲裁完整流程 |
| **只有1种合理做法**（其他方案明显不可行） | ❌ 直接执行，不启动推演 |
| **信息不足无法判断** | ⚡ 追问用户后重新发散 |

**触发时输出启动信号：**

```
═══════════════════════════════════════
 ⚡ [MCTS-TD] 检测到决策需求，启动决策引擎
 触发原因: [具体原因]
 启动模式: [完整推演/简化推演/再推演]
═══════════════════════════════════════
```
