# MCTS-TD Planner — 多方案独立推演决策引擎

> **核心能力**：当有多个候选方案时，每个方案都在内部独立走完完整的执行路径推演（不实际执行），然后汇总比较选出最优方案，最后才执行。

此 Skill 将 **MCTS（蒙特卡洛树搜索）** 和 **TDL（时序差分学习）** 的算法思想注入到 Claude 的推理过程中。它不是实际的数值计算引擎，而是**将算法的核心决策逻辑转化为结构化的推理规则**，让 Claude 在面对复杂决策时，能够像拥有"内部推演棋盘"一样，系统地、逐个地推演每个候选方案，再综合比较。

> 本项目受 [hrpan/tetris_mcts](https://github.com/hrpan/tetris_mcts) 启发，该项目的 MCTS-TD 混合架构在俄罗斯方块游戏中取得了超越人类水平的成绩。

## 三引擎决策流程

```
                    ╔══════════════════════════════════════╗
                    ║      三引擎决策流程                   ║
                    ║      每个引擎有可验证产出，不能跳过    ║
                    ╚══════════════════════════════════════╝

用户需求理解（先听懂用户说了什么）
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  发散引擎 — 头脑风暴（可多轮循环）                            │
│                                                             │
│  ① 六维地图: 技术栈/架构/业务/安全/运维/体验 → 识别盲区       │
│  ② 六路侦查: 项目代码+技术文档+竞品+用户视角+失败案例+趋势    │
│  ③ 视角轮盘: 从不同视角生成跨维度方案（4~8个）                │
│  ④ 追问收敛: 缺信息? → 问用户 → 去掉不可行的                   │
│                                                             │
│  ⚠️ 产出: 方案清单（含六维覆盖度）                            │
│  ⭐ 阶段3必须暂停，用户确认方案列表后才进入推演                  │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  推演引擎 — 子 agent 并行推演                                │
│                                                             │
│  子 agent A（不知道B/C存在）                                  │
│  → 独立推演 → 发现缺信息 → 记录问题 → 继续推演               │
│                                                             │
│  子 agent B（不知道A/C存在）                                  │
│  → 独立推演 → 发现缺信息 → 记录问题 → 继续推演               │
│                                                             │
│  ★ 问用户不阻塞其他子 agent                                  │
│                                                             │
│  ⚠️ 产出: N份推演报告（N=方案数，少一份就是跳过）             │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  仲裁引擎 — 汇总与决策                                       │
│                                                             │
│  ① 汇总推演结果                                              │
│  ② 汇总待解决问题 → 统一问用户                                │
│  ③ 获取答案 → 重新评估受影响方案                              │
│  ④ 执行前自检 → 质疑推演结果                                  │
│  ⑤ 盲区审计 → 检查视角覆盖是否全面                            │
│  ⑥ 输出最优方案 + 执行计划                                   │
│                                                             │
│  ⚠️ 产出: 决策报告（必须有自检、盲区审计和决策结论）            │
└─────────────────────────────────────────────────────────────┘
```

## 验证规则

确保三个引擎都执行了，没有跳过：

- ✅ 有"发散阶段"和"收敛阶段"的记录 → 发散引擎已执行
- ✅ 推演报告数量等于方案数量 → 推演引擎已执行
- ✅ 有决策报告（含自检和盲区审计）→ 仲裁引擎已执行

## 三种运行模式

| 模式 | 适用场景 | 流程 |
|------|---------|------|
| **完整模式** | 方案数 ≤ 5 | 枚举方案 → 逐轮推演 → 汇总比较(CLT-UCB) → 执行 → TD更新 |
| **快速推演** | 方案数 > 5 | 直觉粗筛 → 保留 top-3~5 → 逐轮推演 → 汇总比较 → 执行 |
| **再推演** | 执行途中遇到意外 | 记录TD误差 → 重新推演剩余方案 → 切换执行 |

## ⭐ 阶段性输出规则（用户可见 + 可干预）

> **核心原则**: 发散引擎的每一步产出必须**先展示给用户**，用户确认方案列表后才进入推演引擎。
> 这不是"问用户怎么选"，而是"让用户看到你在想什么"——方案你来推演和决策，但用户有权在看到方案列表后说"再加一个XX视角的方案"或"方案C不用推演了"。

### 输出节奏

```
用户需求理解
    │
    ▼
┌─ 阶段1: 六维领域地图 ──────────────────────────────┐
│  输出: 【六维领域地图】                               │
│  内容: 6个维度的评分 + 盲区识别 + 补资料优先级         │
│  等待: 用户看到后自然继续（不阻塞）                    │
│  用户可干预: "维度X你评低了" / "维度Y不用考虑"         │
└────────────────────────────────────────────────────┘
    │
    ▼
┌─ 阶段2: 六路侦查报告 ──────────────────────────────┐
│  输出: 【六路侦查报告】                               │
│  内容: 各路径发现 + 交叉验证结论                      │
│  等待: 用户看到后自然继续（不阻塞）                    │
│  用户可干预: "路径X你没查" / "这个发现不对"            │
└────────────────────────────────────────────────────┘
    │
    ▼
┌─ 阶段3: 视角轮盘 + 方案列表 ────────────────────────┐
│  输出: 【视角轮盘选择】+【方案列表】                    │
│  内容: 激活了哪些视角 + 每个方案的结构化描述            │
│  ⚠️ 此处暂停，等待用户确认方案列表                     │
│  用户可干预:                                           │
│    "可以，继续推演" → 进入推演引擎                     │
│    "加一个XX视角的方案" → 补充方案后重新展示            │
│    "方案C不用推演了" → 移除方案C后继续                 │
│    "方案A和B合并" → 合并后继续                        │
│    "直接做方案A，不用推演了" → 跳过推演，直接执行       │
└────────────────────────────────────────────────────┘
    │ 用户确认
    ▼
┌─ 阶段4: 推演 + 仲裁 ───────────────────────────────┐
│  推演引擎: 逐方案独立推演（可并行）                    │
│  仲裁引擎: 汇总比较 + 自检 + 盲区审计                  │
│  输出: 【决策报告】                                   │
│  内容: 方案排序 + 自检结论 + 盲区审计 + 执行计划       │
│  用户可干预: "选方案B" / "再想想" / "直接做"          │
└────────────────────────────────────────────────────┘
    │
    ▼
  执行
```

### 关键规则

| 规则 | 说明 |
|------|------|
| **阶段1~2不阻塞** | 六维地图和六路侦查是"信息展示"，输出后自然继续，不停下来等用户 |
| **阶段3必须暂停** | 方案列表展示后，**必须等待用户确认**才能进入推演引擎 |
| **用户跳过权** | 用户说"直接做"→ 跳过所有推演，直接执行最优方案 |
| **用户追加权** | 用户说"加一个XX方案"→ 回到视角轮盘，补充后重新展示 |
| **用户删除权** | 用户说"方案X不用推演"→ 从列表中移除，继续推演其余方案 |
| **推演过程不打断** | 推演引擎启动后不逐步展示（太啰嗦），推演完成后一次性输出决策报告 |
| **只有1个方案时** | 如果视角轮盘只生成1个可行方案 → 跳过推演，直接问"就这个方案，执行吗？" |

### 阶段3 确认提示格式

```
────────────────────────────
 【方案列表确认】

 以上是发散引擎产出的 X 个方案（覆盖 Y/6 个领域维度）。

 接下来将对每个方案进行独立推演，预计需要 Z 轮推演。

 请确认:
   ✅ "继续" / "可以" / "推演吧" → 进入推演引擎
   ➕ "加一个XX视角的方案" → 补充方案
   ➖ "方案X不用了" → 移除方案
   ⚡ "直接做方案X" → 跳过推演，直接执行
 ────────────────────────────
```

## 启动规则

本 Skill 设置为 `alwaysApply: true`，每次对话都会加载。但加载 ≠ 每次都启动完整三引擎流程。

**每次收到用户消息后，Skill 强制启动「发散引擎」进行头脑风暴：**

### 第1步：需求拆解

用户一句话里可能包含多个独立的需求。**拆解的核心方法**：

```
拆解方法：找"方案选择点"

用户说 → 找出所有可以独立选择方案的地方 → 每个"选择点"就是一个子需求

示例1:
  "帮我做一个博客系统，支持markdown，还要有评论功能"
  方案选择点分析:
    → 博客框架选什么？        → 子需求A: 博客系统框架选型
    → Markdown怎么渲染？      → 子需求B: Markdown渲染方案
    → 评论用什么方案？        → 子需求C: 评论系统方案

示例2:
  "这个页面加载太慢了，优化一下"
  方案选择点分析:
    → 慢在哪里？              → 先定位瓶颈（可能要追问用户）
    → 前端优化还是后端优化？   → 子需求A: 优化方向选择
    → 缓存方案用什么？        → 子需求B: 缓存策略（如果有缓存环节）
    → CDN要不要上？           → 子需求C: CDN部署方案

示例3:
  "帮我写一个用户登录功能"
  方案选择点分析:
    → 认证方式选什么？        → 子需求A: 认证方案（JWT/Session/OAuth/…）
    → 前端用什么做？          → 子需求B: 前端实现方案
    → 密码怎么存？            → 这个一般没得选（只能用bcrypt/argon2），不算独立选择点

拆解原则：
  ✅ 按"能不能独立选择方案"来拆
  ❌ 不要硬拆成最小粒度（"登录功能"不需要拆成"用户名输入框"和"密码输入框"）
```

**拆解产出格式：**
```
用户需求: [原文]

拆解结果:
  → 子需求A: [描述] — 可选的方案有？
  → 子需求B: [描述] — 可选的方案有？
  → [如果无法拆解] 这是单一需求，直接进入发散
```

### 第2步：真实发散

对每个子需求（或单一需求），从多个角度生成可行方案。**发散的具体方法**：

```
发散方法：从这些角度逐一过一遍，看哪些角度有多个选择

┌──────────────────────────────────────────────────────────────────┐
│  角度1: 技术栈路线                                              │
│  这个需求可以用哪些不同的技术来实现？                              │
│  → 框架选择: React vs Vue vs Svelte vs 原生JS                     │
│  → 语言选择: Python vs Go vs Java vs TypeScript                   │
│  → 数据库选择: MySQL vs PostgreSQL vs SQLite vs MongoDB           │
│  → 协议选择: REST vs GraphQL vs gRPC vs WebSocket                 │
│  → 注意: 只在有选择权的时候才发散，项目已定的技术栈不重复选         │
├──────────────────────────────────────────────────────────────────┤
│  角度2: 实现策略                                                │
│  这个需求可以用不同的"做法"来实现？                                │
│  → 自己实现 vs 用第三方库 vs 用云服务/SaaS                       │
│  → 从零开始 vs 基于现有框架/模板                                │
│  → 一次性做完 vs 分阶段迭代                                     │
│  → 手动实现 vs 用低代码/生成式工具                              │
├──────────────────────────────────────────────────────────────────┤
│  角度3: 架构方案                                                │
│  这个需求放在架构的哪个层次？                                    │
│  → 纯前端做 vs 纯后端做 vs 前后端配合                            │
│  → 客户端做 vs 服务端做                                         │
│  → 单体 vs 微服务 vs 无服务器                                   │
│  → 同步处理 vs 异步/消息队列                                    │
├──────────────────────────────────────────────────────────────────┤
│  角度4: 设计取舍                                                │
│  这个需求不同的"trade-off"方向？                                  │
│  → 性能优先 vs 可维护优先 vs 开发速度优先                        │
│  → 安全严格 vs 用户体验便捷                                     │
│  → 通用灵活 vs 定制精简                                         │
│  → 本地优先 vs 云端优先                                         │
├──────────────────────────────────────────────────────────────────┤
│  角度5: 外部约束（过滤用，不是发散源）                            │
│  → 有没有不能用/必须用的东西？                                   │
│  → 有没有性能/安全/合规要求？                                   │
│  → 这些不产生新方案，但用来砍掉不合适的方案                      │
└──────────────────────────────────────────────────────────────────┘
```

**关键规则：**

| 规则 | 说明 |
|------|------|
| **不凑数** | 有多少种真实可行的方案就列多少种，1种就1种，10种就10种 |
| **不虚构** | 每一种方案都必须是你实际会去执行的，不能为了数量编造不存在的做法 |
| **不遗漏** | 如果确实存在多种合理方案，不要因为"麻烦"而只列1种 |
| **一需求一发散** | 多个子需求时，每个子需求独立发散，独立列方案清单 |

### 第3步：收敛与决策

```
收敛方法：
  1. 对每个方案过一遍"外部约束"（角度5）
  2. 砍掉明显违反约束的方案
  3. 保留的方案就是"可行方案列表"
```

| 收敛结果 | 行为 |
|---------|------|
| **≥2 个可行方案** | ✅ 进入推演引擎，启动推演→仲裁完整流程 |
| **只有1种合理做法** | ❌ 不启动推演，直接执行。执行后更新知识库（记录"这个场景只有1种做法"） |
| **信息不足无法判断** | ⚡ 追问用户 → 获得信息后回到第2步重新发散 |

**触发时输出启动信号：**

```
═══════════════════════════════════════
 ⚡ [MCTS-TD] 检测到决策需求，启动决策引擎
 触发原因: [具体原因]
 启动模式: [完整推演/简化推演/再推演]
═══════════════════════════════════════
```
