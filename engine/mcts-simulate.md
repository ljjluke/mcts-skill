# 第2步: 逐轮独立推演（核心机制）

> **一句话**: 每个方案在"隔离环境"中独立走完完整的执行因果链，不交叉、不比较、不串扰。
> 推演引擎是整个 MCTS-TD 决策系统的核心——这里产出的推演报告质量，直接决定决策质量。

---

## 核心原则

```
★ 强制规则: 每一轮推演只聚焦一个方案，不交叉、不比较、不串扰 ★

这意味着:
  - 推演方案A时，假设方案B/C/D不存在
  - 推演方案B时，不参考方案A的推演结果
  - 所有方案的推演在"隔离环境"中独立完成

★ 用户可以随时中止:

  在任何阶段，用户说以下任意一句话，立即中止推演流程，进入执行阶段:
  - "直接做"、"别推了"、"不用想了"、"就这个吧"、"随便"
  - 中止时已推演完的方案仍然保留评分
  - 如果有多个方案已推演完 → 选当前评分最高的执行
  - 如果一个都没推演完 → 选第一个方案执行（不推演了，直接干）
```

## 推演前知识注入（含召回步骤）

在开始每个方案的推演之前，**必须执行完整的多路召回流程**，从知识图谱中获取相关知识。

知识图谱由 [td-learner.md](./td-learner.md) 管理。

### 召回执行步骤

```
步骤1: 确定当前查询向量
  特征向量: 从当前任务提取的六维特征
  上下文: {tech_stack, framework, project_type}
  方案关键词: 从方案描述中提取的关键词列表（用于标签匹配）

步骤2: 执行五路召回
  路径1: 精确匹配 → 基于六维特征向量
  路径2: 主维度匹配 → 基于 task_type + domain
  路径3: 领域匹配 → 基于 domain + risk_level
  路径4: 标签匹配 → 基于方案关键词 vs 知识条目 tags
  路径5: 关联扩散 → 基于已召回条目的关联关系

步骤3: 合并评分排序
  去重 → 按综合评分排序 → 取 top-5

步骤4: 输出召回报告
  "召回报告: [具体引用的知识条目K00X, K00Y]
   路径来源: [精确匹配/标签匹配/...]
   冲突提示: [如果有矛盾条目，输出冲突信息]
   引用建议: [推荐优先参考哪条]"
```

### 知识注入到推演

```
根据召回结果，按以下方式注入推演：

case 召回 top-5 中最高分为 CONFIRMED 且 score > 0.8:
  → 作为主参考知识
  → Step 1: "根据历史经验(K00X)，这类任务的典型路径是..."
  → 推演可以紧密参考该知识的上下文

case 召回 top-5 中存在技术栈匹配的知识:
  → 优先参考技术栈匹配的知识（即使它是PROVISIONAL）
  → Step 2: "当前技术栈下，参考K005的经验..."

case 召回 top-5 中存在冲突:
  → 输出冲突提示
  → 推演中分别引用两个方向，注明"历史经验有分歧，需特别注意"
  → Step 3: 最终价值以当前推演为主，历史价值仅作参考

case 召回结果为空或最高分 < 0.2:
  → "无相关知识，冷启动推演"
  → 完全依赖当前分析
  → 推演完成后创建新的HYPOTHESIS条目
```

### 推演报告中的知识引用格式

每个推演步骤中引用的知识必须清晰标注：

```
Step 1: 起点动作
  "如果选择这个方案，第一步要做什么?"
  → 引用: K005 "FastAPI参数校验修复" (score: 0.72, 技术栈匹配)
  → 借鉴: 该方案在同样技术栈下的处理方式是...

Step 2: 关键路径  
  → 引用: K001 "Go/gin空指针修复" (score: 0.95, 技术栈不匹配)
  → 借鉴: 问题模式相同但技术栈不同，需调整具体实现
  → 差异点: Python的Pydantic校验方式与Go的结构体校验不同

Step 3: 终点评估
  → 综合召回知识的加权价值和当前推演
  → 最终价值 = 推演价值 × w1 + 召回加权价值 × w2
  → w1/w2 由召回结果的可信度决定
```

## 推演框架（含缺口检测位）

每个方案的推演遵循固定的"3步因果链"结构。**每个Step完成后必须执行缺口检测**：

```
═══════════════════════════════════════════════════════
  推演方案: [方案名称]
  知识注入: [引用了哪些历史知识/价值函数条目]
═══════════════════════════════════════════════════════

Step 1: 起点动作
  "如果选择这个方案，第一步要做什么?"
  → 具体描述第一个操作（修改哪个文件、调用什么API等）
  → 预期即时结果: [成功/失败/部分成功]
  → 难度评估: [低/中/高]
  
  缺口检测: 
  □ 依赖的信息是否完整?
  □ 涉及的技术是否都理解?
  □ 是否存在未验证的假设?
  → 如果有缺口 → 暂停 → 补全 → 再继续

Step 2: 关键路径
  "第一步做完后，接下来会遇到什么? 如何处理?"
  → 描述执行路径上的关键节点
  → 识别潜在的障碍或风险点
  → 如果出问题，备用策略是什么?
  → 是否引用历史模式? [K00X的路径 / 新模式]
  
  缺口检测:
  □ 路径上的每个节点的上下文是否清楚?
  □ 备用策略的技术可行性是否确认?
  □ 是否依赖了"可能已经变化"的历史知识?
  → 如果有缺口 → 暂停 → 补全 → 再继续

Step 3: 终点评估与项目匹配度分析
  "这个方案走完，最终结果会是什么?"
  → 最终结果描述
  → 遗留问题/副作用
  → 回滚成本（如果失败）
  
  ★ 项目匹配度评估:
  ┌────────────────────────────────────────────┐
  │ 一个价值评分高的方案不一定适合这个项目。      │
  │ 必须评估方案的"优势项"和"劣势项"              │
  │ 与当前项目的实际需求是否匹配。                │
  └────────────────────────────────────────────┘
  
  方案分析:
  优势项/亮点: [这个方案的3个核心优势]
    1. [优势]
    2. [优势]
    3. [优势]
  
  劣势项/代价: [这个方案的3个核心劣势]
    1. [劣势]
    2. [劣势]
    3. [劣势]
  
  项目需求匹配:
  □ 优势1是否正好解决了项目的核心痛点? → [是/否]
  □ 优势2是否正好解决了项目的核心痛点? → [是/否]
  □ 劣势1项目是否在意? → [在意/不在意]
  □ 劣势2项目是否在意? → [在意/不在意]
  □ 劣势3项目是否在意? → [在意/不在意]
  
  匹配度结论:
  优势匹配度: [高(≥2个优势匹配) / 中(1个匹配) / 低(0个匹配)]
  劣势容忍度: [高(≥2个劣势不在意) / 中(1个不在意) / 低(0个不在意)]
  综合匹配度 M = 优势匹配度 × 0.5 + 劣势容忍度 × 0.5
  
  缺口检测:
  □ 终点评估是否有充分依据?
  □ 副作用分析是否覆盖了所有受影响模块?
  □ 回滚成本是否基于实际了解?
  □ 项目匹配度分析是否基于已知的项目约束?
  → 如果有缺口 → 暂停 → 查全局补全箱 → 没有再补全 → 再继续

─── 推演评分 ───
  原始价值 V (0.0~1.0): [纯技术推演评分]
  项目匹配度 M (0.0~1.0): [方案与项目的契合度]
  最终价值 V_final = V × 0.6 + M × 0.4: [综合评分]
  方差 σ² (0.0~1.0): [分数]
  信心水平: [高/中/低]
  关键风险: [列出1-3个]
  全局补全箱引用: [G001, G002, ...]
```

## 推演深度控制

| 任务复杂度 | 推演步数 | 每步深度 |
|-----------|---------|---------|
| 简单（单文件修改） | 2步 | 粗略 |
| 中等（跨文件修改） | 3步 | 详细 |
| 复杂（架构变更） | 3步 | 非常详细 |
| 调试（问题排查） | 2步 | 详细 |

## ⚠️ 推演中的缺口检测与信息补全（关键）

人脑在推演过程中如果发现信息不全，会主动去获取信息再继续。Claude Code 不能脑补，必须**停下来、检测缺口、补全信息、再继续推演**。

### 缺口的两种类型

```
缺口类型1: 需求/上下文信息缺失
  表现: 
    - "用户没有说用什么技术栈" → 不能假设是"最常见的那个"
    - "不知道这个项目的代码结构" → 不能假设"应该是MVC架构"
    - "不清楚用户偏好的风格" → 不能假设"用统一风格"
  
  行动: 暂停推演 → 向用户提问 → 得到回答 → 继续推演

缺口类型2: 技术知识不够
  表现:
    - "这个API的用法我不确定" → 不能自己编一个用法
    - "这个框架的某个特性我没用过" → 不能假设"和别的框架一样"
    - "这个错误信息的含义不清楚" → 不能猜测
  
  行动: 暂停推演 → 查资料(内置知识/技术文档/代码库) → 搞懂后继续推演
```

### 规则1：缺口检测（推演每一步完成后必须检查）

```
每完成一个推演步骤后，必须执行缺口检测：

检测清单:
  □ 这个步骤依赖的上下文信息是否都有？
    如果有缺失 → 标记为"需求缺口"
    "推演Step 1完成，但发现不知道项目使用什么数据库，无法评估方案A的影响范围"
    → 暂停推演，执行信息补全

  □ 这个步骤涉及的技术方案是否都理解？
    如果有不懂的 → 标记为"技术缺口"
    "推演Step 2发现需要用到Kubernetes的StatefulSet，但不确定其PVC回收策略"
    → 暂停推演，执行信息补全

  □ 这个步骤的推演是否依赖了"假设"？
    如果是 → 标记为"假设缺口"
    "假设了用户的API都遵循RESTful规范"
    "假设了数据库版本是MySQL 8.0+"
    → 如果假设有50%以上概率不成立，暂停推演并提问确认
```

### 规则2：信息补全行动

```
根据缺口类型执行不同的补全行动：

需求缺口 → 向用户提问:
  格式:
    "────────────────────────────
     【需求补充】
     推演[方案A]时发现信息缺失:
     问题: [具体什么问题]
     影响: [不补充这个信息，推演会产生XX偏差]
     请补充: [用户需要提供的具体信息]
     ────────────────────────────"
  
  要求:
    - 必须问具体问题，不能笼统问"还有什么信息"
    - 一次最多问2个问题（问太多用户记不住）
    - 用户回答后继续推演
    - 如果用户说"你决定" → 标记为"用户授权假设"，继续推演但降低信心

技术缺口 → 自动查资料:
  行动路径（按优先级）:
    1. 利用Claude的内置知识（模型训练数据中的技术知识）
    2. 搜索相关技术文档（通过WebSearch或read-apidocs工具）
    3. 读取项目中的现有代码（如果有，了解已有模式）
  
  格式:
    "────────────────────────────
     【技术补充】
     推演[方案A]时发现知识不足:
     主题: [什么技术点]
     问题: [具体不清楚什么]
     查资料结果: [查到的事实]
     对推演的影响: [修正了之前的哪个假设]
     ────────────────────────────"
  
  要求:
    - 必须明确说出"我之前不确定XX，查了资料后发现YY"
    - 不能只看一篇资料就下结论，至少2个来源交叉验证
    - 如果查不到 → 标记为"高风险假设"，推演中方差+0.2

假设缺口 → 根据置信度处理:
  ┌─ 置信度 ≥ 80%: 继续推演，在报告中标记"假设:XXX"
  ├─ 50% ≤ 置信度 < 80%: 向用户确认
  └─ 置信度 < 50%: 必须向用户确认，不能继续
```

### 规则3：全局信息补全箱（跨方案共享）

整个决策过程维护一个**全局共享的"信息补全箱"**，所有方案的推演共享。

#### 全局补全箱结构

```
═══════════════════════════════════════════════════════
  全局信息补全箱（跨方案共享）
═══════════════════════════════════════════════════════
  
  已补全的知识:
  ┌──────┬──────────┬────────┬──────────┬─────────┬──────────┐
  │ 编号 │ 主题     │ 缺口类型 │ 补全行动  │ 结果摘要 │ 引用方案 │
  ├──────┼──────────┼────────┼──────────┼─────────┼──────────┤
  │ G001 │ 数据库选型│ 需求   │ 问用户   │ MySQL 8.0│ A, B, C  │
  │ G002 │ StatefulSet│ 技术  │ 查资料   │ PV回收策略确认│ A, B  │
  │ G003 │ 技术栈    │ 需求   │ 问用户   │ Go+gin  │ A, D     │
  └──────┴──────────┴────────┴──────────┴─────────┴──────────┘

  待补全的知识:
  ┌──────┬──────────┬────────┬──────────┬─────────┐
  │ 编号 │ 主题     │ 缺口类型 │ 发现方案 │ 计划行动 │
  ├──────┼──────────┼────────┼──────────┼─────────┤
  │ G004 │ 部署方式  │ 需求   │ 方案C    │ 问用户   │
  └──────┴──────────┴────────┴──────────┴─────────┘
```

#### 核心规则：补全前先查全局箱

```
★ 强制规则：在任何补全行动之前，必须先查全局补全箱 ★

正确的流程:
  推演方案B的Step 2 → 发现"StatefulSet的PVC回收策略不确定"
    → 查全局补全箱 → 发现G002已存在 → 直接引用G002的结果
    → 不需要再次查资料
    → 在方案B的报告中标记"引用全局补全箱G002"

禁止的流程:
  推演方案B的Step 2 → 发现"StatefulSet的PVC回收策略不确定"
    → 直接去查资料 ← ❌ 浪费，方案A已经查过了
```

#### 补全箱查询规则

```
补全前查询:

  1. 用"主题关键词"搜索全局补全箱
     → "StatefulSet PVC" → 匹配G002
     → 直接引用，不需要再查

  2. 用"同义词/近似概念"搜索
     → "K8s持久化存储" → 匹配G002（StatefulSet PVC）
     → 确认是同一个知识点，直接引用

  3. 没找到匹配:
     → 创建新缺口编号G00X
     → 执行补全行动
     → 完成后写入全局补全箱
     → 同时标记"引用方案: [当前方案]"

  4. 补全箱中已有但状态是"待处理":
     → 说明其他方案也发现了同样缺口但还没补全
     → 不重复创建，等补全完成后所有方案共享
```

#### 补全后更新规则

```
一次补全，所有方案受益:

  方案A发现G002 → 查资料 → 补全完成 → 写入全局箱
  方案B需要G002 → 查全局箱 → 已存在 → 直接引用
  方案C需要G002 → 查全局箱 → 已存在 → 直接引用

  效果:
    - 1次查资料，3个方案受益
    - 随着方案增多，边际成本递减
```

#### 补全箱的作用域

```
全局补全箱只在当前决策会话中有效。
不同决策会话（不同任务）之间不共享补全箱——因为上下文不同。

跨会话共享的是"知识图谱"（td-learner.md），
不是"临时补全箱"。
```

#### 补全箱与知识图谱的关系

```
信息补全箱（临时）            知识图谱（持久）
─────────────────────────────────────────────────
当前会话有效                  跨会话持久
记录"缺了什么、怎么补的"      记录"什么场景下、什么价值"
用完即弃                      持续积累
避免重复查资料                避免从零推演
```

**视角覆盖追踪器的使用规则**:

```
规则1: 每次生成新方案后更新
  → 方案A生成 → 视角覆盖计数器: ①技术选型 +1
  → 方案B生成 → 视角覆盖计数器: ④安全优先 +1
  → 方案C生成 → 视角覆盖计数器: ②架构设计 +1, ⑤运维优先 +1

规则2: 方案推演前检查视角盲区
  → 如果某个视角还没有任何方案覆盖 → 标记为"盲区"
  → 在推演该方案时提示: "注意，视角③(业务流程)尚无方案覆盖"
  → 供推演者决定是否需要追加视角

规则3: 汇总比较前输出视角覆盖摘要
  → 在汇总表中附加视角覆盖行
  → 帮助决策者理解"我们是从哪些角度看这个问题的"
  → 如果覆盖度 < 60% → 自动触发盲区审计告警

规则4: 跨方案复用视角
  → 如果一个方案天然覆盖多个视角 → 在覆盖计数器中都+1
  → 例如: 方案C(运维优先)也覆盖了架构模式 → ②架构设计 +1, ⑤运维优先 +1
  → 这有助于提高覆盖率，减少盲区

规则5: 视角覆盖追踪器在决策结束后归档
  → 写入推演总结: "本次决策共覆盖 X/6 个领域维度"
  → 不写入知识图谱（仅作会话记录）
  → 用于改进未来的发散策略
```

### 规则4：补全后对推演的修正

```
补全信息获得后，必须明确修正推演：

修正规则:
  "补全前 -> 补全后"

  如果补全的信息与原推演假设一致:
    → 推演结果不变，信心提升
    → Step X标注: "假设已确认(来源:用户/资料)，信心从MED→HIGH"

  如果补全的信息与原推演假设矛盾:
    → 推演结果必须修正
    → Step X标注: "假设修正(原假设:XX, 修正为:YY)，推演结果调整"
    → 如果修正幅度 > 0.3，需要重新评估后续步骤

  如果补全的信息补充了原有缺失:
    → 推演可以更深入
    → 在修正后的基础上继续推演后续步骤
```

### 规则5：防止错误的两种保护机制

```
保护1: 禁止自动补全需求
  任何时候，如果推演需要"用户的意图/偏好/约束"， 
  不能自己脑补 → 必须暂停推演 → 问用户
  例外: 用户明确说了"你决定"

保护2: 对外查资料不能只信一个来源
  查技术资料时:
    - 至少2个独立来源交叉验证
    - 如果2个来源矛盾 → 标记为"有争议的技术点"
    - 如果只有1个来源 → 标记为"单一来源，谨慎使用"
    - 没有来源 → 标记为"未知领域"，推演方差增加
```

## 知识注入的价值函数处理

### 第一步：评估价值函数的可信度

在引用任何历史价值之前，先评估其可信度：

```
价值函数可信度评估:

  1. 访问次数 (n):
     n < 5  → 数据太少，不可信，仅作参考
     5 ≤ n < 20 → 有一定参考价值，但不可盲信
     n ≥ 20 → 统计上可信

  2. 方差 (σ²):
     σ² < 0.05 → 高度一致，可信
     0.05 ≤ σ² < 0.2 → 正常波动，可参考
     σ² ≥ 0.2 → 结果不稳定，需要谨慎

  3. 时效性:
     24小时内更新 → 新鲜
     1周内更新 → 较新
     超过1个月 → 可能过时

  4. 上下文匹配度:
     历史状态的六维特征与当前状态完全匹配 → 高
     部分匹配 → 中
     低匹配 → 低

综合可信度 = 
  LOW(低)  : 仅作灵感参考，不用于评分
  MED(中)  : 作为推演锚点，但保留质疑权
  HIGH(高) : 可信，可作为价值修正的依据
```

### 第二步：价值注入方式

根据可信度决定注入方式：

```
综合可信度 HIGH:
  → 将历史价值作为推演的"预期锚点"
  → Step 1: "历史数据显示这类任务成功率XX，本次推演以此为基准"
  → Step 3: 最终价值 = 推演价值 × 0.5 + 历史价值 × 0.5

综合可信度 MED:
  → 将历史价值作为"参考基线"
  → Step 1: "历史数据显示XX，但需要结合当前上下文验证"
  → Step 3: 最终价值 = 推演价值 × 0.7 + 历史价值 × 0.3

综合可信度 LOW:
  → 仅将历史价值作为"背景参考"，不作为评分依据
  → "过去有类似案例但数据不足，本次推演主要依赖当前分析"
  → Step 3: 最终价值 = 推演价值 × 1.0（不引用历史）
```

### 第三步：推演中发现价值函数不准时的处理

如果在推演过程中发现**历史价值与当前推理明显矛盾**，不能忽视，必须做纠偏：

```
推演中检测到矛盾的触发条件:
  条件A: 推演Step 1的结果与历史价值方向相反
         (历史说成功率80%，但推理发现当前上下文有根本性差异)
  条件B: 推演Step 2发现了历史数据中不存在的风险
         (技术栈变了/依赖更新了/项目结构改了)
  条件C: 推演的每个步骤都显示历史价值明显偏高/偏低
         (连续3步推理结果都与历史价值偏离 > 0.3)

触发任何一条时，执行以下纠偏流程:
  
  1. 暂停推演，输出"价值函数矛盾检测":
     "历史价值: q=0.85 (基于n=30次历史数据)
      当前推理发现: [具体的矛盾点]
      矛盾原因分析: [为什么历史价值不适用于当前场景]

  2. 修正价值函数使用方式:
     → 该条历史价值从HIGH降级为LOW
     → 当前推演不再依赖该历史价值
     → 推演完成后，在TD更新阶段记录这条矛盾

  3. 继续推演（基于当前分析，不依赖历史）:
     "推演继续，不引用该历史价值。
      本次推演结果将作为新基准，用于更新价值函数。"

  4. 推演结束后在报告中标记:
     "价值函数矛盾已记录: [具体条目]，将在TD更新时处理"
```

### 第四步：分步知识注入

在每个推演步骤中，知识注入的方式不同：

```
Step 1 — 起点动作:
  → 注入: 相似任务的历史成功率
  → 检查: 当前起点与历史起点是否一致?
    • 一致 → 可信度高，价值锚点成立
    • 不一致 → 可能走入不同分支，降低历史权重

Step 2 — 关键路径:
  → 注入: 历史决策序列模式（成功路径/失败路径）
  → 检查: 当前路径是否与历史路径匹配?
    • 匹配 → 可以借鉴历史路径的应对策略
    • 不匹配 → "这是新路径"，标记为潜在矛盾

Step 3 — 终点评估:
  → 注入: 历史终点的价值分布
  → 修正: 最终价值 = 推演价值 × w1 + 历史价值 × w2
  → 权重 w1/w2 由前两步的矛盾检测结果决定
    • 没矛盾: w1=0.5, w2=0.5
    • 部分矛盾: w1=0.7, w2=0.3
    • 明显矛盾: w1=1.0, w2=0.0（弃用历史）
```
