# 第1步: 发散引擎 — 六维地图 × 六路侦查 × 视角轮盘

> **一句话**: 先画地图知道缺什么，再派侦察兵补资料，最后用视角轮盘生成跨维度方案。
> 这是头脑风暴的核心——不是凭直觉列方案，而是系统性覆盖所有关键维度。

---

## 发散引擎全流程

```
六维领域地图（知道缺什么）
    ↓ 识别盲区维度
六路侦查（去补什么）
    ↓ 盲区驱动路径选择
视角轮盘（从不同角度看问题）
    ↓ 每个视角生成1个方案
粗筛（仅方案 > 8时触发）
    ↓ 保留 4~8 个跨视角方案
输出给推演引擎
```

## 1.1 六维领域地图（领域熟悉度检测）

> **核心设计**: 领域不是"熟悉/不熟悉"的二元判断，而是六个独立维度的成熟度评估。
> 每个维度决定不同的发散方向——缺少哪个维度的认知，就从哪个维度去补资料。

```
在生成方案之前，先绘制当前领域的六维地图，识别知识盲区。
```


### 盲区对方案生成的影响规则

```
┌─ 盲区处理规则 ──────────────────────────────────────────────────────────┐
│                                                                          │
│  每个盲区维度必须按以下规则处理:                                          │
│                                                                          │
│  规则1: 该维度评分 < 4 (完全空白)                                        │
│    → 必须先补资料，不能跳过                                              │
│    → 补完资料后该维度才有资格生成方案                                    │
│    → 格式: "维度X完全空白，补资料后才能生成该视角的方案"                  │
│                                                                          │
│  规则2: 4 ≤ 该维度评分 < 7 (部分空白)                                    │
│    → 可以先基于已有知识生成方案，但标注"该视角方案需验证"                 │
│    → 同时在资料收集阶段优先补充该维度的信息                               │
│    → 格式: "维度X部分空白，生成的方案标注为'待验证'，同时补充资料"        │
│                                                                          │
│  规则3: 该维度评分 ≥ 7 (已覆盖)                                          │
│    → 直接用于方案生成                                                    │
│    → 每个已覆盖的维度至少贡献 1 个方案视角                               │
│    → 格式: "维度X已覆盖，生成该视角方案"                                 │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

## 1.2 六路侦查（领域资料收集）

> **核心设计**: 不是简单的"查资料"，而是从六个独立情报源收集信息，交叉验证。
> 每路侦查回答一个不同的问题，覆盖不同视角。
> 只有当六路侦查输出汇聚后，才进入方案生成阶段。

### 六路侦查框架

```
每个侦查路径输出一个结构化的"情报摘要"。
如果该路径无可用信息，标注"无情报"而非跳过。
```

```
═════════════════════════════════════════════════════════════════════
  六路侦查执行清单
═════════════════════════════════════════════════════════════════════

□ 侦查路径1: 当前项目（必查）
  回答: "项目现有代码是怎么处理这个问题的?"
  情报来源: 项目代码、配置文件、数据库 schema、路由定义
  输出: 项目现有实现、代码结构、已有模式
  盲区覆盖: →

□ 侦查路径2: 技术文档与最佳实践（必查）
  回答: "行业标准做法是什么?"
  情报来源: 官方文档、技术博客、RFC/规范
  输出: 主流方案、推荐做法、技术约束
  盲区覆盖: → 技术栈

□ 侦查路径3: 竞品与行业方案（推荐查）
  回答: "其他项目或产品是怎么解决这个问题的?"
  情报来源: 开源项目、竞品分析、社区讨论、Stack Overflow
  输出: 备选方案、不同取舍点、行业趋势
  盲区覆盖: → 架构模式、技术栈

□ 侦查路径4: 用户视角（推荐查）
  回答: "最终用户关心什么?"
  情报来源: 用户反馈、issue/feature request、使用场景分析
  输出: 用户痛点、核心价值点、体验期望
  盲区覆盖: → 用户体验

□ 侦查路径5: 失败案例分析（推荐查，高复杂度任务必查）
  回答: "这个领域常见的坑和失败模式是什么?"
  情报来源: 知识图谱中的失败条目、技术博客的"我们为什么放弃X"、已知反模式
  输出: 踩坑记录、反模式、高风险点
  盲区覆盖: → 安全合规、运维部署

□ 侦查路径6: 技术演进趋势（高复杂度任务必查）
  回答: "这个技术在朝什么方向发展? 有什么即将过时的技术?"
  情报来源: 版本发布记录、迁移指南、技术雷达、弃用公告
  输出: 技术生命周期、迁移成本、前瞻性建议
  盲区覆盖: → 全维度
═════════════════════════════════════════════════════════════════════
```

### 侦查路径的选择规则

```
根据任务复杂度和六维地图的盲区情况，选择侦查路径：

任务复杂度:
  简单（单文件修改）:
    → 必查: 路径1(项目代码)
    → 推荐: 无
    → 时间: < 30秒

  中等（跨文件修改）:
    → 必查: 路径1(项目代码) + 路径2(技术文档)
    → 推荐: 路径4(用户视角) 如果涉及用户体验变更
    → 时间: 30秒 ~ 2分钟

  复杂（架构变更/新功能）:
    → 必查: 路径1 + 路径2 + 路径5(失败案例)
    → 推荐: 路径3 + 路径4 + 路径6(技术趋势)
    → 时间: 2~5分钟

  调试（问题排查）:
    → 必查: 路径1(项目代码) + 路径5(失败案例)
    → 推荐: 路径3(社区讨论，看别人怎么解决相同问题)
    → 时间: 1~3分钟

六维地图盲区驱动:
  如果某个维度评分 < 7，优先执行能覆盖该盲区的侦查路径:
    - 技术栈盲区 → 路径2(技术文档)
    - 架构模式盲区 → 路径3(竞品/行业方案)
    - 业务流程盲区 → 路径1(项目代码) + 路径4(用户视角)
    - 安全合规盲区 → 路径2(安全最佳实践) + 路径5(失败案例)
    - 运维部署盲区 → 路径1(现有部署模式) + 路径5(运维故障)
    - 用户体验盲区 → 路径4(用户视角)
```

### 六路侦查输出格式

```
────────────────────────────
 【六路侦查报告】
 任务: [实现用户登录功能]

 [√] 路径1: 当前项目
  发现: 项目已有 gin-jwt 中间件，但只是基础 token 验证
  代码引用: internal/middleware/auth.go:23-45
  结论: 可以在现有基础上扩展，不需要从头造轮子

 [√] 路径2: 技术文档
  发现: JWT 最佳实践建议使用 access token + refresh token 双 token 机制
  来源: RFC 7519 + auth0.com/blog/... 
  结论: 双 token 比单 token 安全，需要引入 refresh 机制

 [√] 路径3: 竞品/行业方案
  发现: casdoor 和 keycloak 都支持 RBAC + OAuth2，但太重
  来源: github.com/casdoor/casdoor, stackoverflow:xxx
  结论: 第三方方案功能过剩，自实现更适合本项目

 [√] 路径4: 用户视角
  发现: 用户反馈多次提到"登录太频繁"、"需要记住我"
  来源: issues: #42, #87
  结论: remember-me 功能是用户刚需

 [√] 路径5: 失败案例
  发现: 之前 K003 项目用单 token 方案遇到 token 泄露无法撤销的问题
  来源: 知识图谱 K003 (DISPUTED)
  结论: 必须有 token 撤销机制，比如黑名单或短过期 + refresh

 [ ] 路径6: 技术趋势 (跳过，中等复杂度不强制)
  原因: 登录功能技术趋势稳定，无需单独侦查

 跨路径交叉验证:
  - 路径2(双token) + 路径5(撤销需求) → 一致推荐: access+refresh token
  - 路径1(已有gin-jwt) + 路径3(不引入keycloak) → 一致推荐: 自实现
  - 路径4(remember-me) → 唯一在用户视角中出现的需求
 ────────────────────────────
```

## 1.3 视角轮盘（方案生成）

> **核心设计**: 不是笼统地"列方案"，而是强制从多个独立视角各生成一个方案。
> 视角 = 你如何看待这个问题。不同的视角看到不同的解决方案。
> 视角覆盖得越全，头脑风暴的广度就越大。

### 十种基本视角

```
每个视角代表一种不同的"看待问题的方式"。方案生成时，至少覆盖 4 种视角。

┌─────────────────────────────────────────────────────────────────────────────┐
│  视角轮盘（10种视角）                                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ① 技术选型视角: "用什么技术?"                                             │
│     → 关注: 选什么框架/库/技术栈                                            │
│     → 产出: "方案A-用XX技术"                                               │
│     → 适合: 技术栈相关的决策                                                │
│                                                                             │
│  ② 架构设计视角: "怎么组织?"                                               │
│     → 关注: 模块划分、层次结构、通信方式                                    │
│     → 产出: "方案B-用XX架构模式"                                           │
│     → 适合: 架构重构、新模块设计                                           │
│                                                                             │
│  ③ 业务流程视角: "流程怎么走?"                                             │
│     → 关注: 状态机、数据流、业务流程                                        │
│     → 产出: "方案C-重新设计业务流程"                                       │
│     → 适合: 业务逻辑复杂的场景                                              │
│                                                                             │
│  ④ 安全优先视角: "怎么保证安全?"                                           │
│     → 关注: 安全风险、合规要求、攻击面                                      │
│     → 产出: "方案D-安全加固方案"                                           │
│     → 适合: 涉及敏感数据、权限控制的场景                                   │
│                                                                             │
│  ⑤ 运维优先视角: "怎么好运维?"                                             │
│     → 关注: 可部署性、可监控性、可恢复性                                    │
│     → 产出: "方案E-运维友好方案"                                           │
│     → 适合: 需要长期维护、高可用的场景                                     │
│                                                                             │
│  ⑥ 用户体验视角: "用户觉得怎样最好?"                                       │
│     → 关注: 使用流程、反馈、容错性                                          │
│     → 产出: "方案F-体验优先方案"                                           │
│     → 适合: 面向用户的功能                                                  │
│                                                                             │
│  ⑦ 性能优先视角: "怎么能最快?"                                             │
│     → 关注: 延迟、吞吐量、资源消耗                                          │
│     → 产出: "方案G-性能优化方案"                                           │
│     → 适合: 性能敏感的场景                                                  │
│                                                                             │
│  ⑧ 最小成本视角: "怎么能最快交付?"                                         │
│     → 关注: 开发成本、复用现有、渐进式                                      │
│     → 产出: "方案H-最小可行方案"                                           │
│     → 适合: 时间紧迫、资源有限的场景                                       │
│                                                                             │
│  ⑨ 激进创新视角: "如果不考虑限制，最理想的方案是什么?"                     │
│     → 关注: 突破常规、理想方案、未来导向                                    │
│     → 产出: "方案I-理想方案（标注取舍）"                                   │
│     → 适合: 需要打破思维定式时                                              │
│                                                                             │
│  ⑩ 反面视角: "如果故意用最差的方案，是什么样?"                             │
│     → 关注: 反模式、常见错误、应该避免什么                                  │
│     → 产出: "反面教材-不推荐的做法（作为警告）"                            │
│     → 适合: 任何场景（识别出不应走的方向很重要）                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 视角选择规则

```
视角不是随意选的，由 六维地图 + 任务类型 共同决定：

规则1: 六维地图决定视角池
  六维地图中评分 ≥ 7 的维度 → 这些维度的视角自动加入视角池
  六维地图中评分 < 7 的维度 → 这些维度的视角在补完资料前暂不激活

  映射关系:
    技术栈评分高 → 激活视角①(技术选型)
    架构模式评分高 → 激活视角②(架构设计)
    业务流程评分高 → 激活视角③(业务流程)
    安全合规评分高 → 激活视角④(安全优先)
    运维部署评分高 → 激活视角⑤(运维优先)
    用户体验评分高 → 激活视角⑥(用户体验)

规则2: 任务类型补充视角
  性能敏感任务 → 加视角⑦(性能优先)
  时间/资源受限 → 加视角⑧(最小成本)
  高度不确定 → 加视角⑨(激进创新)
  任何任务 → 加视角⑩(反面视角，用于识别应避免的方向)

规则3: 视角数量
  最少 4 个视角，最多 8 个视角
  如果视角池 + 补充视角 < 4 → 从高评分维度补足到至少 4 个
  如果视角池 + 补充视角 > 8 → 根据任务优先级压缩到 8 个

规则4: 每个视角生成 1 个方案
  同一视角不生成多个方案（避免微调变体）
  不同视角的方案必须有实质性差异
  如果两个方案太相似 → 合并，重新生成一个不同视角的方案
```

### 方案之间必须有实质性差异

```
方案之间的差异不能只是"参数不同"或"细节不同"。
差异必须在视角层面上体现:

❌ 不合适的差异（只是参数微调）:
  方案A: 用 MySQL
  方案B: 用 PostgreSQL  ← 同属"技术选型视角"，没本质差异

✅ 合适的差异（来自不同视角）:
  方案A: 用现成的 gin-jwt 中间件扩展 (技术选型视角)
  方案B: 自实现 JWT + Redis 黑名单 (安全优先视角)
  方案C: 引入 Keycloak 作为身份代理 (架构设计视角)
  方案D: 简化登录流程，合并注册和登录 (用户体验视角)

如果发现两个方案实际属于同一个视角 → 合并较弱的一个，替换为其他视角
```

## 1.4 粗筛（仅当方案数 > 8 时触发）

```
粗筛方法: 对每个方案做一次快速直觉评估

评估维度:
  - 可行性: 这个方案在当前上下文中是否可行? (0-1)
  - 成本效益: 投入产出比是否合理? (0-1)
  - 风险: 引入新问题或破坏现有功能的可能性? (0-1)

粗筛公式:
  score = 可行性 × 0.5 + 成本效益 × 0.3 + (1 - 风险) × 0.2

保留 top-3~5 个方案进入深度推演阶段。
被淘汰的方案记录在"备用列表"中，如果主方案全部不可行则重新启用。
```
