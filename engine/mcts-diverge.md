# 第1步: 发散引擎 — 六维地图 × 六路侦查 × 视角轮盘

> **一句话**: 先画地图知道缺什么，再派侦察兵补资料，最后用视角轮盘生成跨维度方案。
> 这是头脑风暴的核心——不是凭直觉列方案，而是系统性覆盖所有关键维度。

---

## 发散引擎全流程

```
六维领域地图（知道缺什么）
    ↓ 识别盲区维度
六路侦查（去补什么）
    ↓ 盲区驱动路径选择
视角轮盘（从不同角度看问题）
    ↓ 每个视角生成1个方案
粗筛（仅方案 > 8时触发）
    ↓ 保留 4~8 个跨视角方案
输出给推演引擎
```

---

## 1.1 六维领域地图（领域自适应）

> **核心设计**: 领域维度不由引擎写死，而是根据任务类型**自动适配**。
> 音乐专家看到的是风格/和声/配器，程序员看到的是技术栈/架构/部署。
> **同一套框架，不同的维度名称，覆盖所有领域。**

### 领域识别与维度加载

```
Step 0: 识别领域
  `python scripts/mcts_compute.py identify-domain --message "<用户消息>" --task "<任务描述>"`
  → 返回: {"domain": "music", "label": "音乐创作"}

Step 1: 加载该领域的六维模板
  `python scripts/mcts_compute.py get-dimensions --domain <domain>`
  → 返回该领域的6个维度定义（名称+描述+检测项）

  示例 — 不同领域的维度:
  软件工程: 技术栈|架构模式|业务逻辑|安全合规|部署运维|用户体验
  音乐创作: 风格流派|调性和声|配器编排|情感表达|演奏难度|听众接受
  教育教学: 课标对齐|学生认知|教学方法|教学资源|评测方式|趣味吸引
  法律:     适用法条|判例参考|风险敞口|文书格式|时效管理|客户预期
  设计:     风格定位|视觉层次|配色排版|交互体验|技术约束|用户偏好
  商业:     市场机会|商业模式|执行路径|风险评估|数据驱动|利益相关方
  通用:     资源条件|目标结果|可选路径|风险未知|时效紧迫|人的因素

Step 2: 对每个维度自评(0-10) + 证据 + 盲区识别
  盲区分类: `python scripts/mcts_compute.py classify-blindspot --score <0-10>`
  完全空白(0-3)→补资料后才能生成 | 部分空白(4-6)→可生成但标注"待验证" | 已覆盖(7-10)→直接生成
```

### ⚠️ 学习深度门禁 — 学不够就不能下结论

```
发散引擎在六维地图和六路侦查完成后，进入方案生成前，
必须通过"学习深度门禁"检查:

  `python scripts/mcts_compute.py check-learning-depth
     --domain <domain> --complexity <simple|medium|complex>
     --scores '{"维度1":8,"维度2":6,...}' --coverage '{"dimensions_covered":4,...}'`

  学习深度等级:
    none: 不需要学习（完全熟悉，直接进入方案生成）
    light: 轻度学习（≥3个维度覆盖 + ≥2个路径 + ≥1个来源）
    standard: 标准学习（≥4个维度覆盖 + ≥3个路径 + ≥2个来源）
    deep: 深度学习（≥5个维度覆盖 + ≥4个路径 + ≥3个来源交叉验证）

  门禁判定:
    ✅ 通过 → 可以进入视角轮盘方案生成
    ⛔ 不通过 → 必须回到六路侦查阶段补全缺口
      输出具体缺口: "维度X尚未覆盖" / "侦查路径不足" / "来源交叉验证不够"
      补全后重新检查，通过才能继续

  强制规则:
    - 高专业度领域（法律/软件工程）→ 至少 standard 级
    - 盲区维度 > 3 个 → 至少 standard 级
    - 盲区维度 > 5 个 → 必须 deep 级
    - 不通过就一直卡在侦查阶段，不允许进入方案生成
```

```
## 1.2 六路侦查（领域自适应）

> 侦查路径同样由领域模板决定。不同领域的"查资料"路径完全不同。
> 加载: `python scripts/mcts_compute.py get-recon-paths --domain <domain>`

### 通用侦查路径模板

```
每个领域都有6条侦查路径（2条必查 + 4条按需）:

软件工程: 项目代码|技术文档|竞品方案|用户视角|失败案例|技术趋势
通用领域: 既有经验|权威参考|他人做法|受影响者视角|失败教训|趋势变化

执行规则:
  - 必查路径: 不执行完不允许跳过
  - 推荐路径: 根据任务复杂度选择
  - 盲区驱动: 六维地图中评分<7的维度，优先执行覆盖该盲区的路径
```

### 学习深度强制执行

```
六路侦查执行后，必须通过"学习深度门禁"才能进入方案生成。
（规则见 1.1 学习深度门禁部分）

如果门禁不通过:
  ⛔ 不允许进入视角轮盘
  → 输出缺口清单: "维度X未覆盖" / "侦查路径Y未执行" / "来源交叉验证不足"
  → 回到侦查阶段补全
  → 重新检查
  → 通过后才能继续
```

### 六路侦查输出格式

输出格式（核心字段）:
  【六路侦查报告】任务: [xxx]
  [√/×] 路径1~6: 发现 + 来源 + 结论
  跨路径交叉验证: (路径X+路径Y → 一致/矛盾 → 推荐)
```

## 1.3 视角轮盘（领域自适应）

> 每个领域有10种视角，名称随领域变化。核心不变：4~8种跨视角方案。
> 加载: `python scripts/mcts_compute.py get-perspectives --domain <domain>`

```
不同领域的视角名称示例:
  软件工程: 1技术选型|2架构设计|3业务逻辑|4安全优先|5运维优先|6用户体验|7性能优先|8最小成本|9激进创新|10反面
  通用领域: 1资源最优|2顶层设计|3流程步骤|4风险控制|5执行可行|6人本体验|7效率速度|8最小投入|9理想方案|10反面

视角选择: `python scripts/mcts_compute.py get-activated-perspectives --scores '<JSON>'`
评分≥7的维度自动激活对应视角 → 任务类型补充 → 反面视角总是激活 → 去重 → 边界(4~8)
```

### 方案之间必须有实质性差异（跨领域通用）

```
方案之间的差异不能只是"参数不同"或"细节不同"。
差异必须在视角层面上体现:

❌ 不合适的差异（只是参数微调）:
  方案A: 用 MySQL
  方案B: 用 PostgreSQL  ← 同属"技术选型视角"，没本质差异

✅ 合适的差异（来自不同视角）:
  方案A: 用现成的 gin-jwt 中间件扩展 (技术选型视角)
  方案B: 自实现 JWT + Redis 黑名单 (安全优先视角)
  方案C: 引入 Keycloak 作为身份代理 (架构设计视角)
  方案D: 简化登录流程，合并注册和登录 (用户体验视角)

如果发现两个方案实际属于同一个视角 → 合并较弱的一个，替换为其他视角
```

### 每个方案的描述格式

```
每个方案的结构化描述包含以下字段:

  方案名称: [简短的标识名称]
  主视角: [属于哪个视角]
  一句话描述: [核心思路]
  来源依据: 
    - 六路侦查结论: [具体引用侦查结果]
    - 知识图谱: [如果有历史K00X引用]
  约束检查:
    - 硬约束: [全部满足/违反X条]
    - 软约束: [满足X条/违反Y条]
  六维覆盖:
    - 优势维度: [这个方案最强的2-3个维度]
    - 忽略维度: [这个方案没考虑的维度，留给其他方案覆盖]
  预期复杂度: [小/中/大]
  补充说明: [任何需要说明的边界情况]
```

## 1.4 粗筛（仅当方案数 > 8 时触发）

粗筛评分: `python scripts/mcts_compute.py rough-filter --solutions '<JSON>' --max-keep 5`
score = 可行性×0.5 + 成本效益×0.3 + (1-风险)×0.2, 保留top-3~5

  备用列表: [...] 视角覆盖度矩阵
