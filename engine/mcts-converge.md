# 第3~3.6步: 仲裁引擎 — 汇总比较 × 自检 × 盲区审计

> **一句话**: 所有方案推演完成后，先排序选最优，再自我质疑推演结果，最后检查视角是否覆盖全面，然后才执行。
> 仲裁引擎是决策质量的最后把关——防止"推演对了方向却错了"。

---

## 仲裁引擎全流程

```
汇总比较（CLT-UCB排序）
    ↓ 选出第1名
推演结果自检（找漏洞 + 反向思考 + 风险评估）
    ↓ 检查"推演得对不对"
领域盲区审计（检查视角覆盖度）
    ↓ 检查"有没有漏掉关键维度"
决策结论（通过 → 执行 / 告警 → 补充方案 / 失败 → 重推演）
```

---

## 第3步: 汇总比较（基于多轮 MCTS 迭代结果）

> 推演引擎已完成多轮 MCTS 树搜索，每个方案经过了 n 次独立模拟，价值估计趋于稳定。
> 仲裁引擎现在基于这些**多轮迭代的收敛结果**做最终排序。

### 输入：来自推演引擎的树搜索结果

```
推演引擎输出的每个方案:
  方案A: n=5, V=0.84, σ²=0.03, 信心=高
         最佳路径: Step1-成功 → Step2-路径a → Step3-完成
         主要风险: Step1-失败 (n=2, V=0.30, 概率低但影响大)
  
  方案B: n=4, V=0.76, σ²=0.08, 信心=中
         最佳路径: Step1-成功 → Step2-规避风险 → Step3-完成
         主要风险: Step2 风险规避失败 (n=1, V=0.52)
  
  方案C: n=3, V=0.62, σ²=0.15, 信心=低
         最佳路径: Step1-成功 → Step2-直接执行
         主要风险: 高方差（不确定性大）
```

### 汇总表（多轮迭代版）

```
┌───────┬────┬──────┬──────┬──────┬────────────────────────────────────┐
│ 方案   │ n  │ V    │ σ²   │ 信心 │ 最佳路径/主要风险                   │
├───────┼────┼──────┼──────┼──────┼────────────────────────────────────┤
│ 方案A │ 5  │ 0.84 │ 0.03 │ 高   │ Step1成功→Step2路径a→完成            │
│       │    │      │      │      │ 风险: Step1失败(n=2,V=0.30)          │
│ 方案B │ 4  │ 0.76 │ 0.08 │ 中   │ Step1成功→规避风险→完成              │
│       │    │      │      │      │ 风险: 规避失败(n=1,V=0.52)           │
│ 方案C │ 3  │ 0.62 │ 0.15 │ 低   │ Step1成功→直接执行                   │
│       │    │      │      │      │ 风险: 高方差（不确定性大）            │
└───────┴────┴──────┴──────┴──────┴────────────────────────────────────┘

列说明:
  n    = 该方案被模拟的次数（多轮迭代的累计，越大越可信）
  V    = 所有模拟的平均价值（多轮收敛后的稳定估计）
  σ²   = 价值方差（模拟结果的一致程度，越小越可靠）
  信心 = 基于 n 和 σ² 的综合判定
```

### MCTS 排序法则（替代旧版 CLT-UCB）

```
多轮迭代后，方案已经有了足够的 n（访问次数）和价值估计。
此时排序直接用"收敛后的 V"而不是 UCB（不需要探索奖励了，因为已经充分探索了）。

排序规则:
  
  1. 按 V（多轮平均价值）从高到低排序
  
  2. 如果两个方案的 V 相差 < 0.05:
     → 比较 n（谁被探索得更多，谁更可靠）
     → 比较 σ²（谁方差更小，谁更稳定）
     → 若仍接近 → 选"最佳路径更简单"的方案
  
  3. 如果第1名领先 < 0.03 但 n < 第2名:
     → 增加 2 轮迭代重新评估第1名（它的价值估计还不够稳定）
     → 重新比较
  
  4. 最终输出:
     → 第1名作为推荐方案
     → 附带: 最佳路径、主要风险、信心水平
```

### 收敛判定（确认迭代是否足够）

```
在最终排序前，检查推演引擎的搜索结果是否已经收敛:

✅ 已收敛的条件（全部满足）:
  □ Root 的总访问次数 N ≥ 方案数 × 4  (每个方案平均至少4次模拟)
  □ 排名第1的方案 n ≥ 5
  □ 排名第1的方案 σ² < 0.10
  □ 排名第1和第2的 V 差距 > 0.05（区分度足够）

如果未收敛:
  → 回到推演引擎追加 3 轮迭代
  → 最多追加 2 次（避免无限循环）
  → 如果仍不收敛 → 当前结果作为最终结果，标注"未完全收敛"
```

## 第3.5步: 推演结果自检（关键防错）

在选中方案执行之前，**必须对推演结果做一次自我质疑**。这是防止"推演错了还按错的执行"的最后一道防线。

### 为什么需要自检

```
推演过程本身也可能出错:
  - Claude 推理时漏掉了某个关键因素
  - 推演时依赖了一个错误的假设
  - 某个方案的风险被低估了
  - 价值函数中的历史数据可能已经过时

不自检就去执行 → 错了也不知道 → 错误的价值被TD学习吸收
  → 后续推演越来越不准 → 恶性循环
```

### 自检步骤

```
在"选择第1名作为执行方案"之后，执行之前，做三件事:

① 找漏洞:
  "这个推演有什么可能出错的地方？"
  → 检查推演过程中是否有模糊的判断
  → 检查是否依赖了未经验证的假设
  → 检查是否有被忽略的风险
  
② 反向思考:
  "如果第2名其实比第1名好，可能是什么原因？"
  → 强迫自己从另一个角度看问题
  → 找出第1名的盲区
  
③ 风险评估:
  "如果选错了，最坏的结果是什么？能承受吗？"
  → 如果能承受 → 执行
  → 如果不能承受 → 进入"再推演"模式，或建议用户手动确认
```

### 自检报告格式

```
═══════════════════════════════════════
  推演结果自检
═══════════════════════════════════════

  ① 找漏洞:
    □ 推演中是否有模糊的判断? → [有/无]
      如果有: [具体说明]
    □ 是否依赖了未验证的假设? → [有/无]
      如果有: [具体说明]
    □ 是否有被忽略的风险? → [有/无]
      如果有: [具体说明]

  ② 反向思考:
    如果第2名[方案B]比第1名[方案A]好，可能的原因:
    - [原因1]
    - [原因2]
    这些原因成立的可能性: [高/中/低]
    如果成立，是否改变选择? → [是/否]

  ③ 风险评估:
    选第1名[方案A]的最坏结果: [描述]
    最坏结果的影响程度: [轻微/中等/严重]
    是否能承受? → [能/不能]

  自检结论:
    ✅ 通过 — 推演可靠，可以执行
    ⚠️ 有风险 — 建议用户确认后再执行
    ❌ 未通过 — 重新推演或换方案
```

### 自检未通过时的处理

```
case 自检结论为 ⚠️ 有风险:
  → 输出自检报告
  → 建议用户确认: "我推演的结果是方案A最好，但有几个不确定的点，你要不要确认一下？"
  → 用户确认后执行
  → 用户否决后回到第2步

case 自检结论为 ❌ 未通过:
  → 输出自检报告
  → 自动回到第2步重新推演
  → 如果连续2次自检都未通过 → 输出"这个问题我不太确定，建议你手动决策"
```

### 熔断机制

```
记录推演准确率:
  每次执行后，对比推演结果和实际结果
  推演准确率 = 最近10次推演中误差 < 0.3 的次数 / 10

熔断规则:
  准确率 < 70% → 自动降级为简化推演（2步，不深入）
  准确率 < 50% → 暂停推演，改为直接问用户"你觉得哪个方案好？"
  连续3次准确率 < 50% → 输出"我可能不太擅长这类问题，建议你手动决策"

准确率恢复:
  如果后续5次推演的准确率 > 80%
  → 恢复完整推演模式
```

## 第3.6步: 领域盲区审计（视角覆盖度检查）

> **核心设计**: 在推演结束、执行之前，最后检查一次——我们的头脑风暴是否真正覆盖了所有关键领域？
> 这一步是防止"方案评分很高，但根本方向就错了"的最后防线。

### 为什么需要盲区审计

```
自检（第3.5步）检查的是"方案推演得对不对"，是垂直方向的深度检查。
盲区审计（第3.6步）检查的是"我们漏掉了什么视角"，是水平方向的广度检查。

两者正交:
  自检: "方案A的推演有没有漏洞?"  ← 垂直深度
  盲区审计: "我们是不是整个漏掉了安全视角?"  ← 水平广度

如果只做自检不做盲区审计:
  → 方案A推演得很完美，但方案A本身是纯技术视角的
  → 用户真正需要的是安全视角的方案B
  → 执行了方案A才发现"哦，应该考虑安全的"
  → 浪费！
```

### 盲区审计框架

```
═════════════════════════════════════════════════════════════════════
  领域盲区审计
═════════════════════════════════════════════════════════════════════

第一步: 列出所有已生成方案的视角
  ┌───────┬────────────────────┬─────────────────────────────┐
  │ 方案   │ 主视角             │ 六维覆盖情况                │
  ├───────┼────────────────────┼─────────────────────────────┤
  │ 方案A  │ ①技术选型          │ 技术栈、架构                │
  │ 方案B  │ ④安全优先          │ 安全合规、技术栈            │
  │ 方案C  │ ⑤运维优先          │ 运维部署、架构              │
  │ 方案D  │ ⑩反面视角          │ (反面教材，不直接覆盖)      │
  └───────┴────────────────────┴─────────────────────────────┘

第二步: 对比六维领域地图，识别遗漏维度
  六维地图:
    [√] 技术栈  — 已覆盖 (方案A, 方案B)
    [√] 架构模式 — 已覆盖 (方案A, 方案C)
    [×] 业务流程 — 未覆盖! ← 盲区
    [√] 安全合规 — 已覆盖 (方案B)
    [√] 运维部署 — 已覆盖 (方案C)
    [×] 用户体验 — 未覆盖! ← 盲区

  盲区维度: 业务流程、用户体验

第三步: 对每个盲区，判断是否需要补充方案

  盲区1: 业务流程
    → 是否需要业务流程视角的方案? → [是/否]
    → 判断依据: 
      如果该功能涉及复杂的实体流转、状态机、多步骤流程 → 是
      如果只是一个简单的CRUD操作 → 否
    → 结论: [需要补充 / 不需要，不影响决策质量]

  盲区2: 用户体验
    → 是否需要用户体验视角的方案? → [是/否]
    → 判断依据:
      如果是面向用户的功能 → 是
      如果是纯后端/基础设施 → 否
    → 结论: [需要补充 / 不需要，不影响决策质量]

第四步: 盲区补充决策

  情况1: 所有盲区都不需要补充
    → ✅ 视角覆盖完整，进入执行阶段
    → 格式: "所有关键维度已覆盖，盲区审计通过"

  情况2: 有需要补充的盲区，且当前排名第一的方案存在明显视角偏科
    → ⚠️ 盲区审计告警，需要补充方案
    → 回到视角轮盘，对盲区维度生成新方案
    → 新方案生成后追加推演

  情况3: 有需要补充的盲区，但排名第一的方案已经覆盖较好
    → 💡 盲区审计提示，但不强制补充
    → 在最终输出中标注: "注意: 方案A没有覆盖XX视角，如果需要可以追加"
    → 用户看到提示后决定是否补充
```

### 盲区审计输出格式

```
────────────────────────────
 【领域盲区审计】
────────────────────────────

第一步: 方案视角汇总
  方案A (①技术选型) → 覆盖: 技术栈, 架构
  方案B (④安全优先) → 覆盖: 安全合规, 技术栈
  方案C (⑤运维优先) → 覆盖: 运维部署, 架构
  方案D (⑩反面视角) → 不直接覆盖

第二步: 六维覆盖度
  [✓] 技术栈    ← 方案A+B
  [✓] 架构模式  ← 方案A+C
  [✗] 业务流程  ← 未覆盖! 
  [✓] 安全合规  ← 方案B
  [✓] 运维部署  ← 方案C
  [✗] 用户体验  ← 未覆盖!

第三步: 盲区评估
  盲区1 - 业务流程:
    是否需要? → 是 (登录功能涉及多步骤流程)
    影响: 高 — 如果业务流程设计不合理，技术再强也没用
  盲区2 - 用户体验:
    是否需要? → 是 (面向用户的功能)
    影响: 中 — 虽然不是核心，但会影响用户满意度

第四步: 补充决策
  方案A (技术选型) V_final=0.88 — 没有覆盖业务和体验视角
  方案B (安全优先) V_final=0.81 — 没有覆盖业务和体验视角
  方案C (运维优先) V_final=0.72 — 没有覆盖业务和体验视角

  → 所有排名靠前的方案都偏技术侧，业务和体验视角空白
  → ⚠️ 盲区审计告警: 建议补充业务流程视角和用户体验视角的方案
  → 追加方案后重新推演比较

  如果用户确认不补充:
  → 最终输出标注: "未覆盖视角: 业务流程、用户体验"
  → 执行时注意: 方案A执行过程中需额外关注业务流程的合理性
────────────────────────────
```

### 盲区审计的触发条件

```
盲区审计在任何情况下都执行，但深度不同:

简单任务（单文件修改）:
  → 快速盲区检查: 30秒
  → 只检查"是否漏掉了明显应该考虑的视角"
  → 格式: "快速盲区检查通过"

中等任务（跨文件修改）:
  → 标准盲区审计: 1~2分钟
  → 执行完整的四步盲区审计
  → 格式: 完整盲区审计报告

复杂任务（架构变更/新功能）:
  → 深度盲区审计: 2~3分钟
  → 执行完整的四步盲区审计
  → 每个盲区维度还追加: "为什么这个维度在本次任务中重要/不重要?"
  → 格式: 完整盲区审计报告 + 追加分析

调试（问题排查）:
  → 重点检查"问题根源定位"的视角是否全面
  → 是否只从一个角度排查了问题?
  → 格式: "排查视角覆盖检查"
```

### 盲区审计 vs 自检对比

```
第3.5步 自检:                     第3.6步 盲区审计:
"方案推演得对不对?"               "视角覆盖得全不全?"
├─ 找推演漏洞                     ├─ 找遗漏视角
├─ 检查假设                       ├─ 检查覆盖度
├─ 反向思考                       ├─ 跨方案对比六维地图
└─ 风险评估                       └─ 盲区补充决策

两者互补:
  自检通过 + 盲区审计通过 → 可以执行 ✅
  自检通过 + 盲区审计告警 → 补充方案后再执行 ⚠️
  自检告警 + 盲区审计通过 → 用户确认后执行
  两者都告警 → 必须重新推演 ❌
```

## 执行遇到意外时的处理

```
执行过程中如果遇到推演时未预见的问题:
  1. 记录意外: 推演时的假设是什么? 实际情况是什么?
  2. 评估影响: 这个意外是否导致当前方案不可行?
     → 如果不可行: 触发"再推演"模式
     → 如果仍可行: 继续执行，但调整推演评分
  3. 再推演: 回到推演引擎，对排名第2的方案做推演（如果之前没做过）
  4. 切换执行: 如果第2名更优，切换过去
```

## 再推演模式的完整规则

```
触发条件:
  执行过程中遇到推演未预见的致命问题，导致当前方案不可行。

再推演规则:

① 是否保留原有推演结果？
  - 原第1名方案的推演结果 → 保留但不作为执行依据，标记为"执行中失败"
  - 原第2名及之后的推演结果 → 保留，作为再推演的参考输入
  - 如果第2名之前已经推演过 → 直接复用其推演报告，无需重新推演
  - 如果第2名之前未推演过（如粗筛时被淘汰）→ 从备用列表中恢复

② 子 agent 能否看到之前的推演记录？
  - 可以看到已失败的方案A的推演和实际失败原因（避免重复犯错）
  - 不能看到其他方案的推演评分（避免锚定效应）
  - 但可以看到其他方案的"方案描述"（知道还有其他选项）

③ 如何合并结果？
  情况1: 第2名已有完整推演报告
    → 直接比较: 原第2名 vs 实际失败原因
    → 如果第2名不受新问题影响 → 切换执行第2名
    → 如果第2名也受影响 → 触发全局方案重评估

  情况2: 第2名只有粗筛结果（未深度推演）
    → 对第2名执行快速推演（2步，不深入）
    → 如果快速推演显示可行 → 执行
    → 如果快速推演也显示有问题 → 从备用列表中选择

  情况3: 所有方案都受影响
    → 回到发散引擎，基于新发现的约束重新生成方案
    → 此时知识图谱中记录"该场景下原方案全部不可行"

④ 再推演后的知识更新
  - 原第1名的实际失败原因 → 更新到知识图谱（作为失败模式）
  - 再推演过程中发现的新约束 → 更新到约束清单
  - 如果再推演成功 → 记录完整决策轨迹到知识图谱
  - 如果再推演也失败 → 触发熔断机制
```

## TD误差记录（执行中）

```
每次遇到意外时:
  TD_error = 实际结果 - 推演预期
  这个误差会被记录，在最终TD更新阶段使用
```

## TD更新与知识写回（执行后）

> 这是 MCTS-TD 中 "TD" 的具体落地——执行完成后，将实际结果与推演对比，更新知识图谱。

### 完整写回流程

```
执行完成后，必须执行以下 TD 更新流程：

步骤1: 计算实际价值
  V_actual = 执行结果的评分:
    +1.0 — 完全成功，没有意外，效果超出预期
    +0.8 — 成功，有一些小调整但总体顺利
    +0.5 — 部分成功，主要目标达成但有妥协
    +0.2 — 勉强完成，但有不少问题
     0.0 — 失败，需要重新开始

步骤2: 计算 TD 误差
  对于 MCTS 树搜索中最优方案的节点路径上的每个节点:

  V_predicted = 节点在 MCTS 树中的 V 值（多轮迭代收敛后的值）
  TD_error = V_actual - V_predicted

  示例:
    方案A（推荐）的树搜索结果: V=0.84, n=5
    实际执行结果: V_actual=0.80
    TD_error = 0.80 - 0.84 = -0.04  ← 推演略乐观，差距不大

步骤3: 遍历最优路径上的节点，逐个更新知识图谱
  for node in [Root, 方案A, Step1-成功, Step2-路径a, Step3-完成]:
    ① 查询知识图谱中是否有匹配条目
    ② 如果匹配:
        更新 n, q, σ²（Welford）
        更新巩固分
        检查是否需要状态转换
    ③ 如果没有匹配 且 |TD_error| > 0.2（差距较大，值得记录）:
        创建新 HYPOTHESIS 条目
        记录: {节点描述, V_actual, 上下文}

步骤4: 更新 MCTS 决策序列模式
  如果整体成功:
    记录执行路径到知识图谱的"成功模式":
    "方案A的路径: Step1成功→Step2路径a→Step3完成 → 实际V=0.80"
  
  如果整体失败:
    记录失败路径和失败原因:
    "方案A的路径: Step1成功→Step2路径a→失败了→原因:XXX"

步骤5: 执行知识图谱特有的维护操作
  - 检查是否有条目超过30天未使用 → 降为 SLEEPING
  - 检查是否有条目超过90天未使用 → 移入 ARCHIVED
  - 检查是否有矛盾知识 → 标记 DISPUTED
```
